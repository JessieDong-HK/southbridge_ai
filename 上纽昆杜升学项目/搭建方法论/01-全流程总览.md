## 阶段一：原始资料收集

### 1.1 业务知识来源

| 来源 | 格式 | 内容 | 怎么获得的 |
|------|------|------|------|
| 服务手册 | PDF（34MB）→ 文字版 | 公司介绍、服务内容、团队信息 | 用户自己写好后导出为PDF |
| 课程大纲 | 6个文档 | 深度版/标准版 × 两校/上纽/昆杜 的详细课时安排 | 用户从内部资料导出 |
| 过往案例 | 口头描述 | 5个学生的录取结果和辅导过程 | 用户在对话中口述 |
| 工作流程 | 口头描述 | 客户接待、督学小助手、自媒体运营等SOP | 用户在对话中口述 |

**🔧 可复用性**：通用。任何新业务都有类似的"核心文档+隐性知识"结构。关键是认识到：
- 正式文档（手册、大纲）通常有错误和遗漏——需要跟实际执行者交叉验证
- 隐性知识（口述的SOP、案例、经验）往往比正式文档更有价值

### 1.2 外部信息来源

| 来源 | 格式 | 规模 | 获取方式 |
|------|------|------|------|
| 昆杜官网 | JSON爬虫 | 680页 + 791图 | 网站爬虫（非本次搭建） |
| 上纽官网 | JSON爬虫 | 7071文件 | 网站爬虫（非本次搭建） |
| 上纽公众号 | txt | 998篇 | 公众号内容提取（非本次搭建） |
| 昆杜公众号 | txt | 780篇 | 公众号内容提取（非本次搭建） |
| 其他博主 | txt | 7份 | 手动收集 |

**🔧 可复用性**：通用——扩展新学校时，官网爬虫是最重要的外部数据源。需要注意：
- 爬虫质量取决于网站结构（JS渲染的页面内容会丢失）
- 最好在爬虫完成后立即做一次内容审计，避免后续才发现大规模缺失

### ⚠️ 这阶段踩过的坑

1. **PDF直接给AI读不可靠**：先用工具提取文字再导入，比让AI读PDF效果好
2. **服务手册内容有错**：市场宣传用的手册 ≠ 实际操作文档，需要以课程大纲等实操文件为准来修正
3. **爬虫数据≠可用数据**：680页里近一半（~327/680）内容少于50字，大量是空壳页面。一开始就要做审计

## 阶段二：架构设计

### 2.1 设计知识库结构

**做了什么**：
1. 确认知识库的使用者（自己 / 新员工 / AI agents）
2. 确认核心使用场景（新员工 onboarding / AI 按需加载 / 日常查阅）
3. 设计文件拆分方案
4. 设计编号和命名规范

**核心产出**：8个一级板块 + 二级文件的树形结构

详细决策过程见 [02-架构设计原则](./02-架构设计原则.md)

**🔧 可复用性**：通用。架构设计是投入产出比最高的阶段——花30分钟想清楚结构，省掉后面无数次返工。

### ⚠️ 这阶段踩过的坑

1. **最开始想用向量数据库**（SQLite + Qdrant + embedding），后来意识到对于"几千字的文档集合"，简单的 Markdown 文件 + 目录结构就够了。技术方案要匹配实际规模。
2. **过早优化**：花了时间设计复杂的 tagging 和 frontmatter 体系，但最后发现好的文件命名和目录结构就解决了90%的发现需求

## 阶段三：业务知识库搭建

### 3.1 从服务手册提取内容

**输入**：《服务手册一字不差文字版.md》（~20页中文）

**做了什么**：
1. AI 通读全文，识别出所有知识主题
2. 按预设的8个板块分配内容
3. 逐个生成文件，每个文件聚焦一个主题
4. 交叉验证：用课程大纲修正手册中的不准确信息

**产出**：18个核心文件（公司概览、两校简介、6个服务项目、课程体系、5个案例、品牌调性、4个SOP）

**🔧 可复用性**：通用。任何"从现有文档建知识库"的任务都可以用这个流程。

### 3.2 修正手册与大纲之间的矛盾

**发现的问题**：
- 手册说辅导是"6小时"，但实际课程大纲加起来远超6小时（含破冰、故事挖掘、文书沟通、AI面试4节课）
- 深度版和标准版的服务内容差异不够清晰

**怎么处理的**：
1. 以课程大纲（实操文件）为准修正服务内容描述
2. 制作深度版vs标准版对比表，标注Cheryl参与度、督学小助手、上课形式等差异
3. 在多个相关文件中保持一致（服务项目 → 课程大纲 → 公司概览）

**🔧 可复用性**：通用。任何有"官方文档"和"实际操作"不一致的地方都需要这个步骤。原则：以实操为准。

### 3.3 创建cross-reference体系

在文件之间建立链接，使AI agent可以按需跳转：
- 服务项目/ai面试.md → 08-昆杜官方资源库.md（面试需要DKU素材）
- 08-昆杜官方资源库.md → 02-两校简介.md（资源库汇总 → 学校定位）
- 各课程大纲 → 对应服务项目（大纲是服务的具体教案）

**🔧 可复用性**：通用。cross-reference 是做"load-on-demand"的关键——不是把所有内容塞进一个文件，而是让文件之间能互相找到。

## 阶段四：外部资料整合

### 4.1 昆杜官网爬虫 → 结构化资源库

**输入**：680页JSON官网爬虫

**做了什么**：
1. 分析爬虫内容分布（什么有价值、什么是噪音）
2. 设计8个专题文件（按使用场景，不是按网站结构）
3. 3个并行 agent 深读 40+ 个高质量页面
4. 按专题组织内容，每个文件有明确的"用途"标注
5. 建立与主知识库的 cross-reference

**产出**：9个文件（~56KB）

完整案例拆解见 [03-外部资料整合](./03-外部资料整合.md)

**🔧 可复用性**：高度可复用。给新学校做同样的事时，这是最核心的步骤。

### 4.2 上纽官网爬虫 → 结构化资源库（第二次，验证方法复用性）

**输入**：2569个页面JSON（英文1818 + 中文750），约1.4GB

**与昆杜的关键差异**：
- 上纽爬虫质量显著更好——50.6%的页面超过200字（昆杜仅~35%），学生故事中位字数981
- 独特内容类型：互动专题(is/)166篇、学生博客(blogs/)18篇、教师著作(works-by-faculty)
- 上纽的核心卖点不同于昆杜：NYU全球网络 + 上海城市优势，而非"文理学院+科研机会"

**做了什么**：
1. 分析爬虫内容分布——发现质量好于预期，设计空间更大
2. 设计8个专题文件——**不照搬昆杜的8个文件结构**，而是根据上纽特质调整：
   - 新增 `03-NYU全球网络与海外学习`（昆杜完全没有对应板块——上纽最强差异化）
   - `02` 改为强调核心课程体系而非"学生画像"
   - `05` 强调上海城市生活而非校园周边
   - `07` 强调 NYU 70万全球校友而非 Duke 20万
3. 并行提取：3个 agent 分工（概览+教育+全球网络 / 研究+校园+毕业 / 学生故事+金句）
4. 编译 + cross-reference + 标注数据充分度
5. 学生故事直接用完整叙事格式（复用昆杜迭代后的标准，避免了两轮返工）

**产出**：9个文件（~45KB）

**🔧 可复用性验证**：
- 昆杜那套"按使用场景设计文件→并行agent提取→编译"的方法论被证明完全可复用
- 第二次做明显更快：因为有昆杜的经验，架构设计、agent分工、输出格式都有了模板
- **关键教训**：结构不要照搬。两所学校核心卖点不同（上纽=全球网络+城市，昆杜=Duke学位+科研），文件结构必须反映各自特质

### ⚠️ 这阶段踩过的坑

1. **Agent 任务过重导致超时**：负责学生故事和金句的 Agent C 主任务在 600 秒后超时——它试图同时统筹读取大量文件。但它的子任务已独立完成了具体文件的读取，内容没丢。教训：**大任务拆成子任务比依赖一个"统筹agent"更可靠**。
2. **第二次做可以跳过"先用简版再深挖"**：昆杜学生故事经历了 bullet point v1 → 完整叙事 v2 两轮迭代。上纽直接用了 v2 标准，一步到位。

### 4.3 昆杜公众号帖子 → 结构化资源库（第三种数据源）

**输入**：780篇公众号txt帖子（2017-2026年，近9年跨度）

**与官网爬虫的关键差异**：
- 数据格式完全不同：官网是JSON结构化页面，公众号是纯txt叙事文本
- 内容性质不同：官网是"官方事实"，公众号是"品牌叙事"——学生故事、招生指南、教育理念长文
- 噪音类型不同：官网噪音是空壳页面（48%不足50字），公众号噪音是宣讲会日程（~219篇）和节日祝福

**做了什么**：
1. 先写总览 README：分析780篇的内容分布、分类统计、与上纽公众号的对比——产出 `昆山杜克大学公众号/README.md`
2. 设计6个专题文件——**与官网资料库结构不同**：
   - 官网资料库是"官方事实骨架"（8个文件：概览、理念、课程、研究、校园、故事、去向、金句）
   - 公众号资料库是"叙事血肉补充"（6个文件：招生申请、教育理念、学生故事、校园体验、毕业生发展、官方声音）
   - 两套资料库互补——公众号不重复官网已有的基础事实，而是补充深度、细节和叙事
3. 并行提取：3个 agent 分工（招生+理念 / 学生故事 / 校园+毕业+金句）
4. Agent 超时后手动接手合成——与4.2的踩坑完全一致

**产出**：7个文件（~114KB）

**🔧 可复用性**：
- 公众号 → 资料库的方法论与官网 → 资料库**核心逻辑一致**（分析分布→按场景设计结构→并行提取→编译）
- 但有两个新增的关键步骤：
  1. **先写总览 README**：公众号帖子数量大、噪音多，先做全局分析写总结文档，既是了解内容的手段，也是后续可独立使用的产出
  2. **明确与官网资料库的互补关系**：公众号资料不是独立存在的，每个文件都要标注"官网已有X，公众号补充Y"
- **结构因数据源而异**：官网资料库8个文件 ≠ 公众号资料库6个文件。文件数量和主题取决于数据源本身的独特价值，不是照搬模板

### ⚠️ 公众号特有的踩坑

1. **Agent 超时问题在公众号资料库上更严重**：3个统筹 agent 全都在600秒超时。公众号帖子是纯文本叙事（不像官网JSON有明确字段），agent 需要读更多内容才能提取。最终采用"子 agent 深读 → 人工接手合成"的混合模式。
2. **就业报告数据嵌在图片里**：2022届和2023届的就业质量报告在公众号以信息图长图发布，txt抓取只能拿到标题和脚注。这是公众号数据源的结构性局限——图片中的数据无法通过文本抓取获得。
3. **学生故事数量太大无法全部深读**：100+篇"她说/他说"系列，15篇深度档案已经足够代表整体。剩下的通过标题分析做速查表。

## 阶段五：质量审计与迭代

### 5.1 爬虫内容完整度审计

**做了什么**：
1. 按字数分布统计（0-9字/10-49/50-199/200-499/500+）
2. 按板块覆盖评估（新闻/中文/研究/校园生活/招生……）
3. 对照已建好的8个文件，评估每个文件的数据充分度
4. 产出审计报告，明确缺口优先级

**关键发现**：
- 48%的文件内容不足50字（几乎为空）
- 新闻板块74%只有标题
- 整个"校园生活"板块没有专属页面，全靠新闻碎片拼凑

完整审计方法论见 [04-质量审计与迭代](./04-质量审计与迭代.md)

**🔧 可复用性**：通用。任何外部数据源都应该在整理后做一次审计，避免"以为有了但其实没有"。

### 5.2 学生故事深度迭代

**输入**：学生故事库 v1（bullet point 版本，每篇3-5行概括）

**做了什么**：
1. 回到7篇完整原文JSON（每篇900-1900词）
2. 提取场景描写、情感弧线、具体对话、感官细节
3. 按"可直接用于视频脚本"的标准重写
4. 增加视频切入点、引用原话、使用场景标注
5. 新增故事速查表和自媒体内容创意

**产出**：学生故事库 v2（内容量 ~3-4x，从"简历摘要"变成"人物特写"）

**🔧 可复用性**：通用。关键模式："先建框架 → 用一段时间 → 发现不够 → 回到源文件深挖"。

## 阶段七：项目总览与自动化路线图

### 7.1 为什么需要这一步

知识库搭建完成后，出现了一个问题：**信息散落在 50+ 个文件和 7 篇方法论文档中，缺少一个统一的"门面"**。

- 自己要看多个文件才能理清项目全貌
- 发给别人看，没有一份文档能一句话说清楚
- 和 AI 对话时，每次都要重新解释背景
- 更重要的是：知识库只是底座，真正的目标是**自动化公司运营**——但这个大方向没有被明确写下来

### 7.2 做了什么

1. **通读全部现有文档**（知识库 + 方法论文档 + README），理解项目全貌
2. **创建 [项目总览.md](../项目总览.md)**——一份涵盖以下内容的综合性文档：
   - 项目定义（一句话 + 一段话）
   - **自动化四阶段路线图**（Phase 1-4，从视频脚本 → 客服转化 → 学生管理 → 人员管理）
   - 已建成的知识库架构（三层信息架构图 + 文件夹地图）
   - 当前进度（已建成 / 待开始）
   - 核心设计原则（6 条摘要）
   - 向不同对象的解释模板（自己 / 同行 / AI agent）
   - 给 AI Agent 的 context 段落（可直接复用）
3. **与项目负责人深度对话**，澄清了：
   - 角色定位：唯一股东 & 老板（耶鲁硕士），非上纽/昆杜校友本人，团队成员才是
   - 自动化优先级：视频脚本（最紧迫）→ 客服转化 → 学生管理 → 人员/实习生管理
   - 内容策略：纯干货路线，不露营销痕迹，靠自然推流；身份定位参考"耶鲁学姐 × 哈佛合伙人"双人工作室模式
   - WeChat 接入目的：电脑不在身边时远程控制 AI
   - 管理瓶颈：目前只能管 3 个人，但有 200+ 份免费实习申请待利用
4. **建立长期记忆体系**：把项目关键信息写入 memory（`项目总览.md` 同目录），使后续 AI 对话自动加载上下文

### 7.3 关键产出

| 产出 | 路径 |
|------|------|
| 项目总览文档 | `项目总览.md` |
| 长期记忆（4 条） | `~/.claude/projects/.../memory/` |
| 更新后的进度看板 | `搭建过程记录/07-日期追踪与计划看板.md` |
| 更新后的全流程总览 | 本文件 |

### 7.4 为什么这一步重要

**知识库是"砖"，项目总览是"建筑图纸"。** 之前只有砖（50+ 个文件），现在有了一张能让人一眼看懂"这栋楼要建成什么样"的总览图。

具体价值：
- **对自己**：自动化路线图明确了"现在该做什么"（视频脚本），避免在知识库里无休止地打磨
- **对他人**：一份文档就能讲清楚项目，不需要翻 7 篇方法论
- **对 AI**：第四节的 context 段落可以直接复用，每次新任务不用重新解释

**🔧 可复用性**：任何知识库项目在搭建完成后，都应该写一份这样的总览。它不是"锦上添花"，而是知识库从"能用"到"好用"的关键一步——否则只有搭建者自己知道怎么用。

## 整个流程的关键经验

### 做对了的
1. **先想架构再写文件**：花30分钟设计结构，避免了后续反复重组
2. **一份文件一个主题**：AI agent 每次只加载需要的文件，而非全部
3. **做完后审计**：建完昆杜资料库马上审计，发现大规模缺失 → 及时调整策略
4. **交叉验证**：服务手册 vs 课程大纲的矛盾被及时发现和修正
5. **按使用场景组织，不按来源结构**：昆杜资料8个文件反映的是"面试要什么""文书要什么"，不是网站的导航结构

### 做对了的（第二轮验证）
6. **结构因校而异，方法论复用**：上纽没有照搬昆杜的8个文件，而是根据"NYU全球网络+上海城市"的特质设计专门结构。但提取和编译的方法完全相同，证明了方法论的有效性
7. **用了第一轮迭代后的标准**：上纽学生故事直接按完整叙事格式写，跳过了 bullet point 阶段，省了一轮返工

### 可以更好的
1. 爬虫完成后应该立刻审计，而不是等整理完才发现大量内容缺失
2. 学生故事应该一开始就用完整原文，第一版的 bullet point 摘要浪费了一轮迭代
3. 架构设计时花太多时间在向量数据库方案上——从小规模开始，需要时再升级
4. **Agent 任务拆解要更细**：上纽 Agent C 统筹任务超时——大任务应拆成独立子任务而非依赖一个 agent 统筹，子任务独立运行更可靠

### 如果要给新学校做同样的事
1. 先拿到所有原始资料（业务手册/课程大纲/官网爬虫）
2. 审计爬虫质量
3. 设计知识库架构
4. 搭建业务知识库
5. 搭建外部资源库
6. 审计 → 迭代
7. 持续维护

预计总耗时（有 AI 辅助）：**3-5 小时**（不含原始资料收集和爬虫时间）
