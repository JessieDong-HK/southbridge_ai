> 最后更新：2026-06-21
>

## 一、客服标准工作流程（4 步 + 签约）

```
Step 1: 添加微信
  家长/学生加微信 → 客服主动发起聊天
      │
      ▼
Step 2: 筛查
  客服询问基本信息 → 判断学生分类
  ├── 优秀
  ├── 过bar
  └── 不满足
      │
      ▼  （不论筛查结果如何，均需在系统上"添加客户"）
      │
Step 3: 开会与发评估
  安排免费评估会议 → 进行会议 → 发送"评估与建议"
      │
      ▼
Step 4: 跟进
  持续跟进沟通 → 推动决策
      │
      ▼
  签约 or 流失
```

## 二、客服与系统交互 · 完整步骤

### 2.1 加微信后 → 筛查阶段（暂不与系统交互）

- 客服加微信后，开始聊天询问基本信息
- 大概率不会立刻在系统上登记，而是筛完甚至聊完才想起来登记
- 登记时只需填学生姓名，后续通过上传截图补全数据

### 2.2 "新建学生"（只需姓名）

客服点击**「新建学生」**按钮，只需填写**学生姓名**即可创建。

> 为什么只需要姓名？实际使用中，客服加了微信之后一般不会立刻登记。等到他想起登记的时候，大概率已经把筛查信息都问完了。此时学生已经在聊天里提供了大量信息，强制填 5 项反而阻碍登记。先建一个空壳，后续上传聊天截图时系统自动补全。

**新建时后端自动创建：**

```
咨询管理/[学生姓名]/
├── 微信聊天记录.md          ← 含三部分空标题
├── 学生信息总结.md          ← 仅标题，待 W1/W2 Agent 填充
├── 客户咨询进度.md          ← 初始 stage: "添加微信"
├── 会议逐字稿/              ← 空文件夹
└── 评估与建议/              ← 空文件夹
```

**新建完成后 — 交互流程：**

```
客服填写学生姓名 → 点击"新建"
         │
         ▼
     后端创建文件夹 + 初始文件
         │
         ▼
     页面弹出提示：
     ┌──────────────────────────────┐
     │  ✅ 已创建学生：阎浩翔         │
     │                              │
     │  ⚠️ 上传前请先在微信中修改备注名：  │
     │    格式：学生姓名 + 关系         │
     │    例如：阎浩翔本人、阎浩翔妈妈    │
     │    否则截图中的昵称无法对应到人    │
     │                              │
     │  📢 请记得上传微信聊天记录截图    │
     │                              │
     │  [ 去上传 ]  ← 点击进入学生详情页 │
     └──────────────────────────────┘
         │
         ▼ 进入学生详情页（不自动打开上传弹窗）
         │  客服可在详情页的"上传更多记录"区域按需上传
```

**学生列表中的提示（方案 A）：**

新建后未上传聊天记录的学生，卡片上显示标签：

```
┌──────────────────────────────┐
│ ⚠️ 阎浩翔                     │
│ 待上传：聊天记录              │
│ 添加微信                      │
└──────────────────────────────┘
```

身份页上传完成 → 对应标签消失。聊天记录上传完成 → 标签完全消失，卡片恢复正常。

| 需要上传 | 去哪儿上传 | 触发哪个 Workflow |
|---------|-----------|-----------------|
| 微信聊天记录截图 | 学生详情页 → 上传区域（选身份） | W1（OCR → 全自动更新） |

### 2.3 后端自动处理（客服无需操作）

提交"新建学生"后，后端自动执行以下操作：

**A. 创建学生文件夹结构**

```
咨询管理/[学生姓名]/
├── 微信聊天记录.md              ← "添加客户"时即创建，含三部分空标题
├── 学生信息总结.md              ← "添加客户"时创建空文件，后续 AI 自动填充
├── 客户咨询进度.md              ← "添加客户"时创建，记录阶段进度
├── 会议逐字稿/                  ← 空文件夹，后续上传逐字稿时存入
└── 评估与建议/                  ← 空文件夹，客服手动点击生成，多场会议合并为一份
```

`微信聊天记录.md` 创建时写入三部分结构头：

```markdown
# [学生姓名] · 微信聊天记录

## 学生家长

## 学生本人

## 家长学生群聊
```

`学生信息总结.md` 创建时为空文件，仅含标题：

```markdown
# [学生姓名] · 学生信息总结
```
（内容由 Agent B 在后续上传聊天记录时逐步填充）

`客户咨询进度.md` 创建时写入初始状态：

```markdown
---
stage: 添加微信
updated: 2026-06-21
---
# [学生姓名] · 咨询进度

## 进度时间线
- 2026-06-21 进入「添加微信」阶段
```

**B. 保存微信聊天记录**

- 同一个学生的所有聊天记录保存在**同一个 Markdown 文件**：`微信聊天记录.md`
- 文件内部分**三个部分**，按身份区分：
  - `## 学生家长` — 客服与家长的单聊
  - `## 学生本人` — 客服与学生的单聊
  - `## 家长学生群聊` — 客服、家长、学生三人都在的群
- 客服上传时选择身份，系统自动追加到对应部分
- 每条记录附带**时间戳**（x年x月x日x点x分）
- 每次上传按消息级别去重，只追加新的

**C. 自动维护两个 AI 文档**

"添加客户"时创建空文件，后续每次上传聊天记录时 AI 自动更新：

| 文件 | 内容 | 更新者 |
|------|------|--------|
| `学生信息总结.md` | 学生的事实信息（成绩、活动、关注点、家庭情况等） | Agent B |
| `客户咨询进度.md` | 只放进度阶段（当前 stage + 时间线） | Agent C |

两个文件客服均不可手动编辑，仅通过上传触发 AI 更新。

## 四、客户详情页（点击客户名字后进入）

客户详情页分为以下模块：

### 4.1 客户咨询进度

- 一个普通的 Markdown 文档
- 由系统通过上传的聊天记录自动生成
- 只放进度信息，不含其他内容
- 每次新上传内容后，自动更新进度
- 界面上不标注"AI 生成"等标签

### 4.2 相关文件

- 显示该学生文件夹里的**所有文件**，有什么就显示什么
- 例如：学生家长微信聊天记录、学生本人微信聊天记录、会议逐字稿、会议总结、评估与建议等
- 所有文件均可**点击查看详情**（均为简单的 Markdown 文档）

### 4.3 上传更多记录

> ⚠️ **上传前请先在微信中修改对方备注名**，格式：学生姓名 + 关系（如"阎浩翔本人""阎浩翔妈妈""阎浩翔爸爸"）。否则截图中的微信昵称无法对应到具体的人。

此区域提供两个上传入口：

**A. 上传微信聊天记录截图**

> ⚠️ 上传前请确保：
> 1. **已在微信中修改备注名**为"学生姓名 + 关系"（如晨昊本人、晨昊妈妈、晨昊爸爸、晨昊姐姐），否则系统无法从截图中的昵称判断谁是谁
> 2. **从最开头开始截图**——滑动到微信提示"你已添加了 XXX，现在可以开始聊天了"，从这句开始截，确保所有对话都被截到
> 3. **截图之间有重复是 OK 的**，只要保证每一句对话都没有遗漏。系统会自动去重

- 选择身份：**学生家长** / **学生本人** / **家长学生群聊**
- 上传截图
- 触发 W1：OCR → 追加到 `微信聊天记录.md` 对应身份的部分（按时间排序，去重追加）

**B. 上传会议逐字稿**
- 选择参会人（可多选）：
  - 客户方：**学生家长**、**学生本人**
  - 公司方：**Cherie**、**Jessie**、**Chloe**
- 上传 TXT → 临时目录 → 触发 W2：预处理去时码 → 保存为 `.md` 到 `会议逐字稿/` → 删除临时 TXT

### 4.4 进度追踪

清晰显示客服标准工作流程的 4 个阶段，当前所处阶段**重点标出**。

进度信息来自 `客户咨询进度.md`，每次上传后 Agent C 自动更新。

**进度显示规则（每个阶段均提取具体时间，含子状态）：**

| 阶段 | 显示内容 |
|------|---------|
| 添加微信 | 加微信时间 |
| 筛查 | 未完成时显示进度（"已问：学校、排名；待问：语言、城市"）；完成时显示完成时间 + 学生分类 |
| 开会与发评估 | 子状态逐步推进：待约定 → 会议时间（评估未发送）→ 会议时间 + 评估已发送时间 |
| 跟进 | 当前轮次 + 子状态（待开始第一轮 / 第一轮进行中 / 第一轮完成待第二轮 / 第二轮进行中 / 两轮均完成） |
| 已签约 | 🎉 + 签约时间 |
| 已流失 | 流失时间 + 拒绝原因 |

- 已完成步骤：显示灰色 + 具体完成信息
- 当前所处阶段：**高亮重点标出**
- 未开始步骤：显示为待定状态

### 4.5 待办显示框

- 系统自动生成的待办事项
- 数据来源：微信聊天记录 + 会议逐字稿
- 每次上传新内容后自动更新
- 客服无需手动编辑
- 界面上不标注"AI 生成"等标签

## 八、Pipeline 阶段定义（与进度追踪对应）

| 阶段 | 含义 | 进入条件 |
|------|------|---------|
| 添加微信 | 已添加微信，尚未开始筛查提问 | 客服上传首个微信聊天记录截图 |
| 筛查 | 正在询问或已完成学生基本信息筛查 | 客服开始询问学校/成绩/语言等（4项） |
| 开会与发评估 | 约会议 → 开会 → 发评估，属于同一个大阶段 | 聊天中提到约会/发评估 |
| 跟进 | 评估已发，进入两轮跟进（内容待设计）：第一轮资源/活动提醒，第二轮学生案例分享 | 评估已发送即进入 |
| 已签约 | 签约成功 | 聊天中明确签约确认 🎉 |
| 已流失 | 家长/学生明确表示拒绝 | 明确拒绝（"不考虑了"等） |

> ⚠️ **已流失 ≠ 长时间未回复**。未回复可能是客服没有及时跟进，不能自动判为流失。无回复的学生保留在"跟进"阶段，由客服手动处理。

## 九、筛查分类说明

| 分类 | 含义 | 后续动作 |
|------|------|---------|
| 优秀 | 三项筛查全部达标，且有突出亮点 | 优先推动 → 约会议 |
| 过bar | 三项筛查基本达标 | 正常推进 → 约会议 |
| 不满足 | 任一项不达标 | 诚实告知风险 → 仅推单模块（但仍在系统中建档） |

> 注意：**不满足**的学生同样需要建档，不能跳过"添加客户"步骤。

## 十、权限约束

> ⚠️ **核心原则：不给客服超出描述的额外权限。**

- 客服通过系统界面上传内容（聊天截图、会议逐字稿）
- 客服**不能直接修改**数据库中的 Markdown 文件
- 系统自动生成的内容（客户咨询进度、待办事项）由系统自动更新，客服不可手动编辑
- 客服只能通过"上传"这个动作来触发系统更新
- 后台管理员（Jess）保留所有直接编辑权限

## 十一、关键业务规则

### 筛查参考（见 `学生筛选检查清单.md`）
- 学校背景：一线城市头部 / 省市重点
- 校内成绩：前 30-40%
- 语言成绩：雅思 6.5+ / 托福 85+ / 多邻国 110+

### 聊天记录存储规则
- 一个学生一个文件（`微信聊天记录.md`），内部分三部分：学生家长 / 学生本人 / 家长学生群聊
- 每条记录含时间戳（x年x月x日x点x分）
- 每次上传按消息级别去重，只追加新的
- 重复截图覆盖到的内容完全OK（去重机制处理）

### 会议相关
- 上传逐字稿时必须选择参会人（客户方 + 公司方，可多选）
- 上传后自动保存到会议逐字稿文件夹
- 上传即触发：更新客户咨询进度 + 更新进度追踪 + 更新待办事项

## 十五、评估建议自动生成 · 设计

### 定位

评估建议是**客户-facing 文档**，和内部咨询记录不同：

| | 咨询记录（内部） | 评估建议（客户-facing） |
|---|---|---|
| **读者** | 内部（Jess、Chloe） | 外部（家长/学生） |
| **目的** | 全面存档 | 展示专业价值 → 推动决策 |
| **内容** | 讨论点、发言、反思 | 匹配度、时间线、突破方向、方案 |
| **语气** | 中性、全面 | 专业、有温度、可微信直接发送 |

### 存放结构

```
咨询管理/[学生名]/评估与建议/
├── 第1场-评估与建议.md
└── 第2场-评估与建议.md
```

### 文档结构

```markdown
# 评估与建议 · [学生名] · 第X场
> 会议日期 | 参会人

## 学生画像速览
## 匹配度分析（上纽 / 昆杜）
## 申请时间线
## 需要突破的方向
## 推荐方案
## 下一步
```

### 自动化方案：上传即生成闭环

```
客服上传聊天截图 / 逐字稿
     │
     ▼ Server 保存文件 → exec('claude workflow run ...') 后台即刻运行
     │
     ▼ AI Workflow 处理（OCR → 分析 → 写入 Markdown）
     │
     ▼ 客服刷新页面 → 数据已在
```
