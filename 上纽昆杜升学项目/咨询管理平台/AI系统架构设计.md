# 南桥客服系统 · AI 架构设计

> 最后更新：2026-06-22（W1 已实现并测试通过）
>
> 回答核心问题：支持蓝图 v2 的客服系统运转，应该如何设计 AI 组件？
> 一个 Agent？多个 Agent？多个 Workflow？Skill？怎么组合？

---

## 一、先厘清概念：你这个系统里有什么可用的

| 构建块 | 是什么 | 适合什么 | 你现在有的 |
|--------|--------|---------|-----------|
| **Skill** | 一份 Markdown 指南，教 Claude 如何完成某类任务 | 人触发的复杂任务，需要判断力 | 3 个：`咨询管理`、`话术引擎`、`视频脚本创作` |
| **Workflow** | 一个 JS 脚本，编排多个 AI 子代理按阶段流水线执行 | 事件驱动的多步骤自动化，可并行 | 2 个：`会后自动化`、`生成评估建议` |
| **子 Agent（在 Workflow 内）** | Workflow 中单次 AI 调用，做一件具体的事 | 流水线中的一环，可返回结构化数据 | 每个 Workflow 里 3-5 个 |
| **独立 Agent** | 一个完全独立的 Claude 实例 | 需要隔离的并行任务 | 0 个（你没用过） |

**核心结论：你不需要"做一个 Agent"。你需要的是 Workflow + 子 Agent 的组合。**

## 三、核心设计决策：按触发事件分组，不按 AI 任务分组

**❌ 错误思路：一个 AI 任务 = 一个 Workflow**
- 会导致 10+ 个 Workflow，互相依赖，触发链复杂
- 比如上传聊天截图要依次触发 A → B → C → D → E → F，6 个 Workflow 排队

**✅ 正确思路：一个触发事件 = 一个 Workflow，内部包含多个子 Agent**
- 只有 2 个主要触发事件：上传聊天截图、上传会议逐字稿
- 每个 Workflow 内部并行跑多个子 Agent
- 和现有 `会后自动化.workflow.js` 的模式一致
- 看板文件的重新生成不是独立的 AI Phase——它们是写入阶段的最后几步，**纯代码聚合数据后写出**，只在数据实际变化时才触发

## 四、Workflow 1：聊天记录处理（上传微信截图 → 全自动更新）

### 触发条件

网页上传微信聊天截图时，**必须选择两个字段**，缺一不可上传：

| 字段 | 交互方式 | 数据来源 |
|------|---------|---------|
| 学生姓名 | 下拉菜单（或勾选列表），不允许自由填写 | 调用后端获取当前所有已建档学生 |
| 身份 | 单选：学生家长 / 学生本人 / 家长学生群聊 | 固定选项 |

完整链路：

```
客服打开上传界面
     │
     ▼ 网页调用 API 获取学生列表 → 渲染为下拉菜单
     │
     ▼ 客服选择学生 + 选择身份 + 选择截图文件
     │
     ▼ 点击上传 → server.js 保存截图 → exec('claude workflow run 聊天记录处理 ...')
     │              触发参数：{ student: "阎浩翔", identity: "学生家长" }
     │              后台异步执行，不阻塞网页响应
     │
     ▼ 即刻运行本 Workflow
```

> **为什么学生姓名必须用选项而不是填写？** 防止客服手写名字造成同一个人多个文件夹（比如"阎浩翔"写成"闫浩翔"）。学生姓名只能是已建档列表中的，不存在的学生无法上传。

### 如何获取学生列表

**方案：扫描 `咨询管理/` 目录，过滤系统文件夹。**

```
读 咨询管理/ 下的所有子目录
     │
     ▼ 过滤掉系统目录：
     │   · _模板/
     │   · 话术库/
     │   · 跟进管理/
     │   · 以 _ 开头的所有目录（预留）
     │   · 以 . 开头的所有目录（预留）
     │
     ▼ 剩余目录名 = 学生姓名列表
```

**为什么不用单独的学生名单文件？**

- 文件系统本身已经是数据库——每个学生一个文件夹
- 维护两份数据（文件夹 + 名单文件）必然产生不一致
- 这套约定已经运转中：现有的 `_模板`、`话术库`、`跟进管理` 和学生名字没有冲突

**API 端点：**

```
GET /api/students
→ 扫描 咨询管理/ → 过滤系统目录 → 返回 ["阎浩翔", "陈亦枫", "陈亦帆", "王紫涵", "黎子诚"]
→ 网页渲染为下拉菜单
```

### 输入参数与聊天记录存储结构

Workflow 接收触发参数 `{ student: "学生姓名", identity: "学生家长|学生本人|家长学生群聊" }`，上传的截图路径由 server.js 提供。

同一个学生的聊天记录，可能来自三种身份：
- **学生家长**：客服和家长的单聊
- **学生本人**：客服和学生的单聊
- **家长学生群聊**：客服、家长、学生三人都在的群

三种身份的聊天记录**保存在同一个 Markdown 文件**里：
```
咨询管理/[学生姓名]/微信聊天记录.md
```
文件内部按身份**分三个部分**：

```markdown
# [学生姓名] · 微信聊天记录

## 学生家长
（家长单聊记录）

## 学生本人
（学生单聊记录）

## 家长学生群聊
（群聊记录）
```

每次上传时，客服选的"身份"决定新消息追加到文件的哪个部分。

### Phase 1：OCR 提取（Agent A 单独跑，必须先完成）

```
┌──────────────────────────────────────────────────────┐
│ OCR提取Agent（W1）                                    │
│   输入：截图图片                                       │
│   输出：结构化聊天消息列表                              │
│         [{speaker: "家长", content: "...",            │
│           datetime: "2026-06-20 14:30"}]             │
└──────────────────────────────────────────────────────┘
```

> ⚠️ **微信截图中的日期是相对的**：微信聊天界面不显示完整日期，只显示"今天""昨天""周三"等。Agent A 在提取时需要：
> - 读取截图文件的创建时间或上传时间作为参考锚点
> - 将"今天"→ 上传日期，"昨天"→ 上传日期-1天，"周三"→ 最近一个周三的具体日期
> - 输出中的 `datetime` 必须是 `YYYY-MM-DD HH:MM` 绝对格式

### Phase 2：写入聊天记录（Agent A 结果落地，纯代码）

```
3a. 更新微信聊天记录文件（必须执行）
    写到：咨询管理/[学生名]/微信聊天记录.md
    前提：该文件已在"添加客户"时创建，内含三部分空标题
    条件：总是执行（Agent A 返回了新消息）
    步骤：
      ① 拿 Agent A 的结构化输出 [{speaker, content, datetime}, ...]
      ② 根据触发参数中的"身份"，将新消息转为 Markdown 行：
         - 每条消息格式：`**[时间]** [说话人]：\n[内容]\n`
         - 不同消息之间空一行
         - 新消息追加到对应身份的部分下面（如身份="学生家长"，则追加到"## 学生家长"下方）
      ③ 读入已有聊天记录文件 → 按消息内容 + 时间戳去重
      ④ 只追加新消息到对应身份的部分（保持时间顺序）
      ⑤ 每条新消息自动附带时间戳（x年x月x日x点x分）
    触发：执行完后 → 标记 chatChanged = true
          → 聊天记录文件已更新，后续 B/C/D/E 均可读取
```

> ⚠️ **为什么写聊天记录作为独立 Phase 2 而不是让 Agent A 自己写？** 因为 Phase 3 的 B/C/D/E 都需要读聊天记录文件。如果 A 的结果不先落地为文件，B/C/D/E 读不到最新聊天内容。所以 Phase 1 → Phase 2 → Phase 3 是严格先后顺序。

### Phase 3：分析并写入（B/C/D/E 并行跑，各自端到端）

四个 Agent 全部读取 Phase 2 刚更新的 `微信聊天记录.md`，各自独立分析，写入不同目标文件。互不依赖，可并行。

```
┌──────────────────────────────────────────────────────┐
│ 信息提取Agent（W1）                                    │
│                                                      │
│   输入：                                              │
│     · 微信聊天记录.md（本学生）                        │
│     · 学生信息总结.md（本学生，文件一定存在，可能仅含标题）│
│                                                      │
│   分析：                                              │
│     从聊天中提取关于这个学生的新事实：                   │
│     · 成绩变化（新SAT/托福/雅思分数、校内排名等）        │
│     · 活动更新（新竞赛、新项目、新奖项）                │
│     · 关注点变化（本来只考虑上纽，现在考虑上纽和昆杜）     │
│     · 家庭新情况（预算变化、家长态度变化）              │
│     只输出"已有总结里没有的新信息"                      │
│                                                      │
│   写入：                                              │

┌──────────────────────────────────────────────────────┐
│ 进度判断Agent（W1）                                    │
│                                                      │
│   输入：                                              │
│     · 微信聊天记录.md（本学生）                        │
│     · 客户咨询进度.md（本学生，读当前 stage）           │
│                                                      │
│   判定规则（内联，控制 token）：                        │
│                                                      │
│   ① 添加微信                                          │
│     条件：仅有添加微信的对话，尚未开始筛查提问           │
│     显示：加微信时间（从聊天第一条消息获取）              │

│   ② 筛查                                              │
│     条件：客服开始询问学生的具体信息                     │

│       · 地理位置（哪个城市）                            │
│       · 学校（哪所高中）                                │
│       · 校内排名（前百分之多少）                        │
│       · 英语语言成绩（雅思/托福/多邻国）                 │

│       · 未问全 → stage = "筛查"，显示"已问：学校、排名   │
│         ；待问：语言成绩、地理位置"                      │
│       · 已问全 → stage = "筛查"，显示"已完成筛查         │
```

### 关键澄清

> **Q: 为什么待办清单和会议安排是全局文件，而不是每个学生一份？**
>
> A: 客服每天打开工作台，一眼看到"今天所有学生有什么待办""接下来有哪些会议"，比一个个点进学生文件夹高效得多。每个学生占一个 `##` 部分，同一个文件里一目了然。

> **Q: 聊天上传为什么也要更新会议安排？**
>
> A: 因为很多会议时间是直接在微信里约定的（"那我们下周三下午3点？""好的"），Agent E 专门从聊天里提取这种信息。如果提到了就更新，没提到就跳过。

## 五、Workflow 2：会议记录处理（上传逐字稿 → 全自动更新）

### 触发条件

网页上传会议逐字稿（TXT 格式）→ `server.js` 保存到临时目录 → exec('claude workflow run 会议记录处理 ...') → 后台异步，即刻运行本 Workflow

触发参数（由网页上传时客服选择 + server.js 生成）：

| 参数 | 来源 | 示例 |
|------|------|------|
| `student` | 客服在网页选择学生（下拉菜单） | `"阎浩翔"` |
| `meetingNumber` | 客服选择场次（第1场/第2场…） | `"2"` |
| `attendees` | 客服多选参会人（Jessie / Chloe / Cherie / 学生家长 / 学生本人，可多选） | `["Jessie", "学生家长", "学生本人"]` |
| `tmpPath` | server.js 自动生成，指向临时 TXT | `"/tmp/transcript_abc123.txt"` |

### Phase 1：定位原始逐字稿（纯代码）

| 读什么 | 从哪读 |
|--------|--------|
| 刚上传的逐字稿 TXT | 临时目录（由触发参数 `tmpPath` 指定） |
| 学生姓名 + 场次 + 参会人 | 触发参数 |

### Phase 2：预处理（纯代码，TXT → 清洗 → 保存为 MD）

```
读入原始 TXT 文件
     │
     ▼ ① 提取会议基础信息（从文件名/参数获取）：
     │     学生姓名、第几场、参会人列表、日期
     │
     ▼ ② 清洗逐字稿正文：
     │     · 删除每行开头的时码（如 "00:15:23"、"[00:15:23]"）
     │     · 删除空行和纯分隔符行
     │     · 保留：说话人 + 说话内容
     │
     ▼ ③ 输出为干净的 Markdown 文件：
     │     咨询管理/[学生名]/会议逐字稿/2026-06-25-第2场.md
     │
     │     文件格式：
     │     # [学生名] · 第X场
     │     > 日期：2026-06-25
     │     > 参会人：Jessie, 学生家长, 学生本人
     │
     │     **Jessie**：今天想和你聊聊...
     │     **家长**：好的，我们主要关心...
     │     **学生**：我的SAT成绩...
     │
     ▼ ④ 标记：预处理完成 → 后续 Agent 全部读取此 MD 文件
     │
     ▼ ⑤ 删除临时 TXT：`rm [tmpPath]`
```

> ⚠️ **为什么要预处理？** 原始 TXT 每行带时码（"00:15:23"），这些时码对 AI 分析无用，大量消耗 token。清洗后只保留"谁说了什么"，后续 Agent 处理效率更高、token 花费更少。
>
> 🗑️ **原始 TXT 为临时文件**：预处理完成、MD 确认无误后，删除临时目录下的原始 TXT。学生文件夹里只保留清洗后的 MD。

## 六、Workflow 3：评估与建议生成（手动触发，非自动）

### 为什么不做成自动的？

- 一场会议可能时间不够，需要多场会议之后才发送评估与建议
- 可能先和家长开会（信息不全面），再和学生开会（补充信息），然后基于全部会议内容生成完整评估
- 发送时机由客服判断，不应自动

### 触发方式

网页上的「📋 评估与建议」Tab → 在对应学生卡片上点击「⚡ 生成」按钮 → POST /api/trigger-assessment → exec('claude workflow run 生成评估与建议 ...') → 后台异步，即刻运行

触发参数：`{ student: "阎浩翔" }`

### Phase 1：读取所有相关资料

| 读什么 | 从哪读 |
|--------|--------|
| 所有会议逐字稿 | `会议逐字稿/*.md`（全部场次） |
| 所有会议总结 | `会议逐字稿/*-会议总结.md`（全部场次） |
| 学生信息总结 | `学生信息总结.md` |
| 已有评估与建议（如有） | `评估与建议/*.md`（用于更新而非覆盖） |

### Phase 2：生成/更新评估与建议

```
读入该学生的全部会议资料
     │
     ▼
  生成 6 节评估文档（若有旧版则更新）：
     ## 学生画像速览
     ## 匹配度分析（上纽 / 昆杜）
     ## 申请时间线
     ## 需要突破的方向
     ## 推荐方案
     ## 下一步
     │
     ▼
  写入：咨询管理/[学生名]/评估与建议/评估与建议.md
       （注意：只有一个文件，多次会议后更新，不按场次分文件）
```

> 多场会议的资料合并生成一份评估，新会议后重新生成即覆盖更新。

### Phase 3：更新客户咨询进度

```
评估文档已生成 → 更新 客户咨询进度.md
  → 子状态："评估已发送（2026-06-28 15:00）"
```

## 八、回答你最核心的问题

> "我应该设计一个 agent，还是多个 agent，还是多个 workflow？"

**答案：2 个新 Workflow，每个 Workflow 内部包含 4-5 个子 Agent。**

- **不要**做一个"全能 Agent"——太臃肿，无法并行，一次 failure 全挂
- **不要**做 10 个微型 Workflow——触发链太复杂，维护成本高
- **要做**：2 个事件驱动的 Workflow（聊天处理 + 会议处理），每个内部按 Phase 组织子 Agent
- **Skill 保持不变**：`咨询管理` 和 `话术引擎` 继续服务人触发的场景

这个设计：
- 和现有 `会后自动化.workflow.js` 的模式**完全一致**
- 触发事件清晰（只有 2 种上传）
- 子 Agent 可并行跑（AI 分析阶段的多个 Agent 互不依赖）
- 看板不单独做一个 Phase（纯代码聚合，条件触发）
- 出问题时可以单独重跑某个 Workflow

### 非 AI 操作：新建学生

`POST /api/students` 是纯后端文件操作，不需要 AI：

```
客服填写姓名 → POST /api/students { name: "阎浩翔" }
     │
     ▼ server.js 执行：
     mkdir 咨询管理/阎浩翔/
     mkdir 咨询管理/阎浩翔/会议逐字稿/
     mkdir 咨询管理/阎浩翔/评估与建议/
     创建 微信聊天记录.md（含三部分空标题）
     创建 学生信息总结.md（仅标题）
     创建 客户咨询进度.md（stage: 添加微信）
     │
     ▼ 返回成功 → 前端弹窗提示上传身份页/聊天记录
```

**架构**：设计文档中的 W1 Workflow 被替换为服务端内置的两段式管线：

```
上传截图
  │
  ▼
Step 1: 通义千问 Qwen-VL-Plus（阿里云百炼）
  → 纯 OCR 读图，提取所有文字
  → 颜色识别：绿色气泡→南桥，白色气泡→对方
  → 保存至 微信聊天记录-OCR原始输出.txt（可排查）
  │
  ▼
Step 2: DeepSeek V4 Pro（Anthropic 兼容格式）
  → 读原始 OCR → 合并碎片 → 修正日期 → 识别说话人
  → 输出结构化 JSON → 追加至 微信聊天记录.md（去重）
  │
  ▼
Step 3: DeepSeek V4 Pro
  → analyzeStudentMaterial：更新学生信息总结 + 客户咨询进度 + 待办清单 + 会议安排
```

**模型配置**：
| 模型 | 用途 | API | 格式 |
|------|------|-----|------|
| DeepSeek V4 Pro | 文本分析（主模型） | api.deepseek.com/anthropic | Anthropic Messages |
| 通义千问 Qwen-VL-Plus | 截图 OCR | dashscope.aliyuncs.com | OpenAI Chat |

**关键发现**：
- DeepSeek V4 Pro 不支持 Vision → 必须外挂视觉模型
- DeepSeek V4 Pro 是推理模型，thinking 消耗 token → max_tokens 需 ≥8192
- Qwen-VL 纯读图 + DeepSeek 文本结构化 的两段式比单模型一步到位更稳定
- 两步各存 txt 文件，出问题可精确排查

**测试结果**：
- ✅ 单张截图：日期、说话人、内容均准确
- ✅ 多张截图：同时上传 2 张，全部正确处理
- ✅ 重复上传：去重生效，无重复消息
- ✅ UI 流程：新建→提醒→必填校验→进度追踪→完成查看，全链路通畅
