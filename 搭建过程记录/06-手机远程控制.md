# 06 手机远程控制 Claude Code

> 把 Claude Code 接入微信，出门在外用手机发消息就能指挥电脑干活。完整记录了两个方案的踩坑与最终选型。

---

## 最终效果

在微信里给 ClawBot 发消息 → 消息桥接到电脑上的 Claude Code → CC 执行任务 → 结果回复到微信。

场景举例：
- 通勤路上发"看看 utils.py 的 KeyError"→ 到公司代码已改好
- 等地铁 5 分钟发"把 notes 整理成周报"→ 到工位拿 8 成完稿
- 午饭排队让 CC 跑回归测试 → 回来直接看报告

## 两个方案对比

| | 方案A：cc-wechat | 方案B：cc-connect ✅ |
|------|------|------|
| 原理 | 依赖 Claude Code 实验性 Channel 功能 | 独立的消息桥接服务 |
| 稳定性 | ❌ 被 Anthropic 服务端灰度控制，部分账号无法使用 | ✅ 不依赖 CC 实验功能，独立运行 |
| 守护进程 | ❌ 需要用 tmux 手动挂后台 | ✅ 内置 launchd 守护进程，开机自启 |
| 多平台 | 仅微信 | 微信 / 飞书 / 钉钉 / Telegram / Discord 等 10+ 平台 |
| 最终结论 | 跑不通，已弃用 | **当前使用方案** |

### 方案A 踩坑记录

`cc-wechat` 的三个命令：
```bash
npx cc-wechat setup      # 扫码绑定微信 ✅ 成功
npx cc-wechat install    # 生成 MCP 配置 ✅ 成功
claude --dangerously-load-development-channels server:wechat  # 启动 ❌ 消息发过去无响应
```

**失败原因**：`--dangerously-load-development-channels` 是 Claude Code 的实验性功能，Anthropic 服务端按账号灰度开放。你的账号不在灰度名单内，所以 WeChat Channel 虽然在本地注册成功，但实际消息通道被服务端拦截，微信发消息石沉大海。

**关键教训**：涉及"实验性功能"的方案，在正式使用前必须先验证消息能否往返。不能因为扫码成功就以为配通了。

## 方案B：cc-connect 完整配置流程

### 架构

```
微信 App → 腾讯 iLink API → cc-connect（独立桥接服务）→ Claude Code → 结果回传微信
```

关键区别：cc-connect 不经过 Claude Code 的 Channel 机制，直接用微信官方 Bot API 收发消息，然后驱动本地 Claude Code 干活。

### 前置条件

| 要求 | 说明 |
|------|------|
| Node.js ≥ 18 | `node --version` 检查 |
| Claude Code | 已安装并登录 claude.ai 账号 |
| 微信最新版 | 支持 ClawBot 插件 |
| 电脑常开 | 关机就停了 |

### 第一步：安装 Node.js（如已安装可跳过）

到 https://nodejs.org 下载安装。

### 第二步：修复 npm 权限（如遇到 EACCES 报错）

```bash
npm config set cache ~/.npm-cache
```

### 第三步：设置用户级 npm 目录（避免 sudo）

```bash
npm config set prefix ~/.npm-global
echo 'export PATH="$HOME/.npm-global/bin:$PATH"' >> ~/.zshrc
```

### 第四步：安装 cc-connect

```bash
npm install -g cc-connect@beta
```

一定要 `@beta` 版本，守护进程功能只有 beta 版本有。

### 第五步：创建配置文件

创建 `~/.cc-connect/config.toml`：

```toml
language = "zh"

[log]
level = "info"
file = "~/.cc-connect/cc-connect.log"

[[projects]]
name = "agent-code"
path = "/Users/jess/Desktop/自动化agent code"

[projects.agent]
type = "claudecode"

[projects.agent.options]
mode = "default"

[[projects.platforms]]
type = "weixin"
```

### 第六步：扫码绑定微信

```bash
cc-connect weixin setup
```

终端弹二维码 → 微信扫码 → 确认绑定。

### 第七步：安装为守护进程

```bash
cc-connect daemon install --config ~/.cc-connect/config.toml
```

这条命令会：
- 注册为 macOS launchd 系统服务
- 开机自动启动
- 后台静默运行，关终端不影响

## 日常管理

| 操作 | 命令 |
|------|------|
| 查看运行状态 | `cc-connect daemon status` |
| 停止服务 | `cc-connect daemon stop` |
| 重启服务 | `cc-connect daemon restart` |
| 查看实时日志 | `cc-connect daemon logs -f` |
| 彻底卸载 | `cc-connect daemon uninstall` |

## 聊天记录位置

所有微信上的对话历史保存在：

```
~/.cc-connect/sessions/agent-code.json
```

在 Finder 中按 `Cmd + Shift + G`，粘贴 `~/.cc-connect/sessions/` 即可访问。

在微信里给 ClawBot 发以下命令：

| 命令 | 功能 |
|------|------|

| `/help` | 查看所有命令 |

## 关键设计决策

### 为什么选 cc-connect 而不是 cc-wechat？

一句话：**依赖实验性功能的方案不是方案。** cc-wechat 的服务端灰度控制意味着你无法确定它什么时候能用、什么时候不能用。cc-connect 用微信官方 iLink API 独立通信，绕过了 Anthropic 的灰度开关。

### 为什么要用守护进程而不是 tmux？

tmux 确实能挂后台，但：
- 电脑重启后需要手动重新拉起
- 不熟悉终端的人容易搞混 session
- launchd 守护进程是 macOS 原生机制，开机自动启，更可靠

🔧 **可复用性**：高。这套流程适用于任何需要在手机端控制本地 AI agent 的场景。核心思路是"独立桥接服务 + 守护进程"，不绑定具体平台。
