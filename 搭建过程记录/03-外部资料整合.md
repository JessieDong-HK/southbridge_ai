# 03 外部资料整合 — 昆杜官网案例

> 这是整个项目中最复杂的单次操作：把 680 页凌乱的网站爬虫数据变成 9 个结构化、按场景组织的文件。
>
> 这一套流程完全可以复制到任何新学校。

---

## 起点：一个 680 页的 JSON 爬虫

### 我们有什么

```
duke_kunshan_official_website_20260603/
└── duke_kunshan_official_site/
    ├── all_pages.json       (682个页面对象)
    ├── pages/               (682个独立JSON文件)
    └── images/              (791张图片)
```

每个页面 JSON 包含：
- `title` — 页面标题
- `text` — 正文全文
- `url` — 原始URL
- `page_type` — 页面类型（news / about / research / zh-hans...）
- `word_count_estimate` — 字数估算
- `images` — 图片URL和本地路径

### 我们要什么

不是"DKU 官网镜像"，而是**可以直接用于升学辅导的资料库**——面试回答素材、文书灵感、家长答疑数据、自媒体内容来源。

---

## 步骤一：分析爬虫内容分布

### 先搞清楚"里面有什么"

不能上来就建文件——680 页里大量是噪音。先做快速摸底：

**按页面类型**：
- `news` — 新闻（约326篇）
- `zh-hans` — 中文页面（约217篇）
- `research` — 研究（约85篇）
- `about` — 关于我们（约35篇）
- 其他 — 导航模板、活动、标签页

**初步判断**：
- 新闻最有价值（学生故事、毕业数据、校园活动）
- 中文页面有价值（中文引用、中国学生案例）
- 研究页面大部分是行政招标公告（没用）
- 导航模板、标签页 → 全部忽略

**🔧 复现步骤**：让 AI agent 读取 `all_pages.json`，按 `page_type` 字段统计分布，列出每个类型的文件数和代表性标题。

### 踩坑：一开始不应该让 AI 逐个文件读

680 个文件一个一个读是浪费时间。正确做法：
1. 先看 `all_pages.json` 的汇总信息
2. 挑 5-10 个代表性页面深读，了解内容质量
3. 基于代表页面设计结构
4. 再批量提取

---

## 步骤二：设计专题结构（不是按网站结构）

### ❌ 不应该这样做

按 DKU 官网的导航结构：
```
Academics / Admissions / Research / Campus Life / News / About...
```

### ✅ 应该这样做

按**辅导业务的使用场景**：

| 使用场景 | 需要什么内容 | 对应文件 |
|------|------|------|
| "Why DKU?" 面试回答 | 学校数据、教育理念、独特优势 | 01-学校概览、02-教育理念 |
| "你想学什么专业？" | 课程体系、Signature Work | 03-学术与课程 |
| "你想跟哪个教授做研究？" | 研究中心、教授名录、学生研究机会 | 04-研究资源 |
| "你能为校园贡献什么？" | 社团、社区、文化 | 05-校园生活 |
| "讲一个你的榜样/受启发的故事" | 学生故事 | 06-学生故事库 |
| 家长问"毕业能去哪" | 毕业生去向数据 | 07-毕业生去向 |
| 文书和面试需要引用原话 | 校长/教授/学生金句 | 08-官方引用 |

### 📌 关键洞察

> **不要按信息来源的结构来组织知识库。按使用者的任务来组织。**
>
> 一个学生在准备 Why DKU 面试时，不会想"我先看看 Academics 页面，再去 About 页面找找使命陈述"。他只会想"DKU 到底哪里吸引我"。
>
> 知识库的组织逻辑 = 使用者的任务逻辑。

---

## 步骤三：并行提取

### 怎么做

启动 3 个 AI agent 并行工作，每个有不同的提取任务：

| Agent | 任务 | 读了哪些页面 |
|------|------|------|
| Agent 1 | 学校数据、排名、录取、学费 | `about__*`, `admissions__*`, 关键 news 页面 |
| Agent 2 | 教育理念、学术体系、研究资源 | `academics__*`, `research__*`, 教授相关页面 |
| Agent 3 | 学生故事、校园生活、毕业数据 | `news__*` 中有完整正文的（约40篇） |

### 为什么并行

- 三个任务之间的内容互不依赖
- 节省时间：3 个 agent 同时读 → 结果同时返回
- 每个 agent 只关注自己的领域，不会混淆

### 关键：给 agent 明确的提取指令

不是"把官网内容整理一下"，而是：

> "提取所有关于 DKU 本科课程体系的信息，包括：专业设置、学分结构、通识教育要求、Signature Work。用中文整理，保留官方数据和英文原名。如果信息缺失，标注'待补充'。"

**🔧 复现步骤**：写好每个 agent 的提取范围、输出格式要求、信息来源标注要求。

---

## 步骤四：编译和连接

### 从 agent 返回结果到最终文件

1. 把 agent 返回的内容按专题分配
2. 统一格式（标题层级、表格样式、引用标注）
3. 标注信息来源（哪个 JSON 文件）
4. 建立与主知识库的 cross-reference

### 具体做了的连接

```
昆杜官网资料/02-教育理念.md
  → 上纽昆杜升学辅导/02-两校简介.md（学校定位）
  
昆杜官网资料/06-学生故事库.md  
  → 上纽昆杜升学辅导/03-服务项目/ai面试.md（面试素材）
  → 上纽昆杜升学辅导/03-服务项目/文书辅导.md（文书素材）
  
昆杜官网资料/04-研究资源.md
  → 上纽昆杜升学辅导/03-服务项目/ai面试.md（第2课"专属资源匹配"）
  
上纽昆杜升学辅导/08-昆杜官方资源库.md
  → 昆杜官网资料/ （快速索引）
```

### 📌 关键洞察

> **不要把外部资料库的所有内容复制到主知识库。** 建一个"快速索引文件"（08-昆杜官方资源库.md）指向外部资料库即可。这样外部资料更新时，不用改主知识库。

---

## 步骤五：内容审计

### 建完之后必须做这一步

详情见 [04-质量审计与迭代](./04-质量审计与迭代.md)

**核心发现**：建完 8 个文件后做了审计，发现：
- 48% 的爬虫文件字数不足 50（几乎为空）
- 新闻板块 74% 只有标题
- "校园生活"文件全靠新闻碎片拼凑，缺少专属页面

这个发现改变了后续策略：不是所有 8 个文件都同等可靠，标注了每个文件的"数据充分度"。

---

## 步骤六：深度迭代

### 不是建完就完了

学生故事库的第一版是 bullet point（每篇 3-5 行）。用起来发现不够——写视频脚本需要画面感和情绪。

于是回到 7 篇完整原文（每篇 900-1900 词），从头提取场景、对话、情绪转折。第二版内容量 ~4x，增加了视频切入点和原话引用。

这个过程本身就是一条有价值的方法论：**先用快速版本建框架 → 用起来发现不够 → 回到源文件深挖。**

## 整个流程的输入输出总结

| 步骤 | 输入 | 操作 | 输出 |
|------|------|------|------|
| 1. 分析内容分布 | 680页JSON | AI统计page_type分布 | 内容地图 |
| 2. 设计专题结构 | 内容地图 + 业务需求 | 人工设计8个文件 | 文件大纲 |
| 3. 并行提取 | 40+个高质量JSON | 3个agent并行深读 | 提取内容 |
| 4. 编译连接 | agent返回内容 | 统一格式+建链接 | 9个markdown文件 |
| 5. 内容审计 | 9个文件 + 680页JSON | 统计缺失+评估充分度 | 审计报告 |
| 6. 深度迭代 | 7篇完整原文 | 逐篇提取场景细节 | 学生故事v2 |

## 复现清单：给新学校做同样的事

假设你要给上纽（NYU Shanghai）做官网资料库：

- [ ] 确认爬虫数据位置和格式（JSON页数、文件结构）
- [ ] 快速浏览10-15个代表性页面，了解内容质量
- [ ] 统计 page_type 分布，找出有价值的内容类型
- [ ] 设计专题文件结构（按辅导场景，不按网站结构）——这步可以复用昆杜的 8 个文件模板
- [ ] 写 3 个 agent 的提取任务描述
- [ ] 并行运行 agent
- [ ] 编译结果 → 统一格式 → 标注来源 → 建 cross-reference
- [ ] 审计：统计各板块覆盖度，标注数据充分度
- [ ] 选 1-2 个板块做深度迭代（如学生故事）

预计时间（有 AI 辅助）：**1-2 小时**

## 方法论验证：上纽官网资料库

昆杜这套流程在上纽身上完整跑了一遍，验证了方法论的可复用性。

### 实际用时

约 **40 分钟**（3 个 agent 并行提取 ~25 分钟 + 编译 ~15 分钟），比第一版的 1-2 小时快了约一半。

### 为什么第二次更快

1. **架构模板可参考**：8 个文件的设计逻辑是一样的，不需要从头想
2. **输出标准已确立**：学生故事直接用完整叙事格式，跳过了 bullet point 阶段
3. **Agent 分工模式验证过**：知道什么任务适合拆成子任务、什么适合单个 agent

### 结构不是照搬的

虽然方法论一样，但上纽的 8 个文件结构与昆杜不同：

| 昆杜有 | 上纽有 | 变化原因 |
|------|------|------|
| 05-校园生活与社区 | 05-上海生活与校园指南 | 上海本身是卖点 |
| 02-教育理念与学生画像 | 02-教育理念与核心课程 | 上纽强调核心课程体系 |
| — | 03-NYU全球网络与海外学习 | 上纽独有板块 |

### 踩坑：Agent 超时

负责学生故事提取的 Agent C 主任务在 600 秒后超时——它试图同时统筹读大量文件。但提前拆出去的子任务独立完成了具体文件读取，内容没丢。

**教训**：大任务拆成独立子任务比依赖一个统筹 agent 更可靠。统筹 agent 在等所有子任务返回时容易超时。

## 方法论第三次验证：公众号 → 资料库（新数据源类型）

### 背景

在上纽官网资料库之后，有了第三种外部数据源：**公众号帖子**（txt格式，纯文本叙事，无结构字段）。

这一次要处理的是昆山杜克大学公众号的780篇帖子（2017-2026年），目标同样是转化为升学辅导可用的结构化资料。

### 与官网爬虫的本质区别

| 维度 | 官网爬虫（前两次） | 公众号帖子（第三次） |
|------|------------------|-------------------|
| 数据格式 | JSON，有 title/text/url/page_type 字段 | 纯txt，无结构字段 |
| 内容性质 | 官方事实（数据、声明、结构介绍） | 品牌叙事（学生故事、申请指南、教育理念长文） |
| 最大价值 | 覆盖面广，系统性强 | 深度强，叙事完整，有原话引用 |
| 主要噪音 | 空壳页面（48%不足50字） | 宣讲会日程（28%）、节日祝福 |
| 文件数 vs 有价值数 | 680页 → ~40篇高质量页面 | 780篇 → ~300篇有实质内容 |

### 新增步骤：先写总览 README

这是公众号资料库搭建中**新增的关键步骤**——在官网资料库的流程里没有这一步。

**为什么需要**：
- 780篇纯文本没有 page_type 字段可以快速统计分类
- 必须先通读标题、抽样正文，才能了解"里面有什么"
- 写 README 的过程本身就是内容分析的过程

**做法**：
1. 启动一个 Explore agent 通读所有文件名（按日期+标题命名），统计年度分布、文件大小分布
2. 按标题关键词做初步分类（招生宣讲会/学生故事/申请指南/校园活动……）
3. 抽样30+篇横跨不同年份的帖子验证分类准确性
4. 写出总览 README（与上纽公众号的 README 对标），包含：总体规模、内容分类、与官网的互补关系、对升学辅导的价值评估、后续处理建议
5. 产出：`昆山杜克大学公众号/README.md`

**这个步骤的价值超出了"准备阶段"**：README 本身就是一个独立可用的产出——"每一次打开这个文档，就一目了然地知道，哦，原来昆山杜克的公众号是这么回事儿"。

### 结构设计：与官网资料库互补而非重复

**核心原则**：公众号资料库不重复官网已有的内容。官网给"骨架"，公众号给"血肉"。

| 官网资料库（8个文件） | 公众号资料库（6个文件） | 关系 |
|---------------------|---------------------|------|
| 01-学校概览与数据 | — | 公众号无新增 |
| 02-教育理念与学生画像 | 02-教育理念与人才培养 | 公众号补充招生院长署名文章 |
| 03-学术与课程 | — | 公众号无新增 |
| 04-研究资源与项目 | — | 公众号极少涉及 |
| 05-校园生活与社区 | 04-校园就读体验 | 公众号补充第一人称叙事 |
| 06-学生故事库 | 03-学生成长故事 | 公众号深度远超官网（100+篇 vs 8篇） |
| 07-毕业生去向 | 05-毕业生发展全景 | 公众号补充就业报告原文和校友追踪 |
| 08-官方引用与金句 | 06-官方声音与品牌表达 | 公众号补充毕业演讲全文和招办声音 |
| — | 01-招生申请实战手册 | **公众号独有**（官网无详细申请指南） |

### 执行：3 Agent 并行 + 手动接手

**Agent 分工**（与官网资料库的模式一致）：
- Agent 1：招生申请手册 + 教育理念与人才培养
- Agent 2：学生成长故事（最大的文件）
- Agent 3：校园就读体验 + 毕业生发展全景 + 官方声音与品牌表达

**遇到的问题**：
- 三个统筹 agent **全部在600秒超时**——比官网资料库更严重（那次只有 Agent C 超时）
- 原因：公众号帖子是叙事文本，agent 需要读更多内容才能提取，合成阶段的信息量远超官网JSON的结构化字段
- 子 agent 的源文件读取全部成功（50+篇帖子的完整内容），内容没丢

**解决方案**（与之前一致但更极端）：
- 子 agent 独立完成文件读取（不依赖统筹 agent）
- 统筹 agent 超时后，人工接手合成——用子 agent 返回的素材直接写文件
- 最终产出 7 个文件（~114KB）

### 这一次验证了什么

1. **"先写 README 再建资料库"是一个可复用的模式**：对于大量无结构文本数据源，先做全局分析写总结文档，既能帮助自己理解数据，又能产出一个独立可用的成果
2. **资料库结构必须因数据源而异**：不能把官网的8文件模板套到公众号上。要问"这个数据源独特提供了什么"，然后据此设计结构
3. **互补性标注很重要**：每个文件开头标注"官网已有X，本文件补充Y"，让使用者知道什么时候看哪个
4. **Agent 超时是公众号数据源的系统性挑战**：叙事文本的信息密度远高于结构化数据。对于这种数据源，"子 agent 深读 + 人工合成"的混合模式可能比纯 agent 统筹更可靠
5. **图片数据的丢失是公众号 txt 抓取的结构性局限**：就业报告等信息图的数据无法从 txt 提取，需要在数据收集阶段就意识到这一点

### 复现清单：公众号 → 资料库

假设要处理另一所学校的公众号帖子：

- [ ] 确认帖子文件数量和命名格式
- [ ] 启动 Explore agent 做全局分析：年度分布、文件大小分布、按标题初步分类
- [ ] 抽样 20-30 篇验证分类，了解内容质量
- [ ] **先写总览 README**（总规模、分类、与已有资源的关系、价值评估）
- [ ] 确认该公众号与已有官网资料库的互补关系——哪些是独特的、哪些是重复的
- [ ] 设计专题文件结构（按公众号的独特价值，不照搬官网模板）
- [ ] 写 3 个 agent 的提取任务描述（特别注意：子任务要拆细，避免统筹 agent 超时）
- [ ] 并行运行 agent
- [ ] 如有超时，人工接手合成（子 agent 返回的素材已足够）
- [ ] 审计：标注数据充分度，特别是图片数据的缺失

预计时间：**1-1.5 小时**（含 README 编写 ~20 分钟 + agent 提取 ~30 分钟 + 合成 ~20 分钟）
