# 05 Prompt 模板库

> 实际验证过的 AI 协作模式。不是完整的逐字 prompt（太长了），而是模板 + 关键参数 + 为什么这样写。

---

## 模式一：从源文档搭建知识库

### 场景
你有一个长文档（如 20 页服务手册），想拆成结构化的知识库。

### Prompt 模板

```
我有一份业务文档，需要把它拆成一个结构化的知识库。

## 知识库结构（预设）
[在这里列出你设计好的文件结构，每个文件标注它应该包含什么]

## 源文档
[贴上文档内容]

## 要求
- 每个文件只写它该写的内容，不要跨文件重复
- 如果源文档中某个文件的内容缺失，标注"[待补充]"
- 如果源文档的不同部分对同一件事有矛盾，以[权威来源]为准
- 使用中文，保留英文专有名词原名
- 每个文件开头标注"信息来源：[来源名称]"
```

### 关键参数
- **预设结构**：在让 AI 做之前，先自己设计好文件结构——AI 擅长"按框架填充"，不擅长"设计框架"
- **矛盾处理规则**：明确告诉 AI 以哪个来源为准
- **标注缺失**：让 AI 主动说"这里缺内容"，比假装有更好

### 实际效果
这个模式用于从《服务手册一字不差文字版.md》生成了 18 个业务知识库文件。

## 模式二：并行深读 + 按场景提取

### 场景
有一批 JSON 爬虫网页（几十到几百个），想从中提取特定主题的内容。

### Prompt 模板（给 agent）

```
你的任务是阅读 DKU 官网爬虫文件中与 [主题] 相关的页面，提取以下内容：

## 提取范围
- 只读 files with page_type = [news / about / academics...]
- 只提取 word_count > 200 的页面内容
- 重点关注包含 [关键词] 的页面

## 提取内容
1. [子主题1] — 包括具体的名称、数据、描述
2. [子主题2] — 包括原话引用、具体案例
3. [子主题3] — 包括可量化的数据

## 输出格式
- 中文叙述，保留英文专有名词和引用的英文原文
- 每条信息标注来源文件名（如 news__xxx.json）
- 缺失的信息标注"[未在爬虫数据中找到]"
- 如某类信息只在1-2个页面出现且不完整，标注"[来源有限]"
```

### 为什么要限定 word_count
3 个 agent 并行工作，如果不限定——它们会浪费时间去读那些只有标题的空壳页面。直接告诉它"只读字数>200的"，效率提升巨大。

### 实际效果
这个模式用于从 680 页 DKU 爬虫中提取 8 个专题文件。3 个 agent 并行跑了约 10-15 分钟。

## 模式三：内容审计

### 场景
已经建好知识库，想全面了解原始数据的完整度。

### Prompt 模板（给 agent）

```
在 [爬虫目录] 中，做一次完整的内容审计：

## 维度一：文件字数分布
统计所有 JSON 文件按字数分档的数量和占比：
- 0-9 字 / 10-49 / 50-199 / 200-499 / 500+

## 维度二：按板块覆盖评估
对于 [已有知识库的 8 个文件]，逐一评估：
- 这个文件的信息来源是否充分？
- 爬虫中是否缺少了该主题的专门页面？
- 有哪些内容是从二手来源拼凑的？

## 维度三：最大的内容缺口
列出爬虫中完全缺失或严重不足的板块。

## 维度四：对照评估
把审计结果和我们建好的 [X 个文件] 对照，标注每个文件的数据充分度（⭐1-5）
```

### 实际效果
审计发现 48% 文件字数<50、校园生活零专门页面、新闻 74% 只有标题。这些发现改变了后续策略——不是所有 8 个文件都同等可靠。

## 模式四：深度叙事提取

### 场景
有完整的原文（如学生故事），但当前版本只是 bullet point 摘要，需要提取生动的叙事细节。

### Prompt 模板（给 agent）

```
从 [具体文件路径] 中，提取可用于视频脚本创作的具体细节：

## 提取维度
1. **具体瞬间** — 什么时候、在哪里、发生了什么？描述场景
2. **情绪旅程** — 害怕过什么？什么让 ta 惊讶？哪个时刻改变了一切？
3. **感官细节** — 画面、声音、气味、具体地点
4. **对话式引用** — 像真人说话的原话，不是 PR 金句
5. **具体数字** — 金额、日期、数字等增加质感的细节
6. **挑战和失败** — 什么出错了？什么是难的？怎么挣扎过？
7. **转折点** — 事情发生变化的那个时刻
8. **意外的人性化细节** — 怪癖、意外、好笑的事、小小的善意
9. **关系** — 具体谁帮助了 ta？怎么帮助的？
10. **前后对比** — 来的时候是什么样，走的时候是什么样？
```

### 关键：为什么列这 10 个维度
bullet point 摘要的问题是只说"结论"——"她做了三个社团的主席"。视频脚本需要"过程"——"她最初在候补名单上，不甘心，反复找宿管要反馈，终于拿到 RA 位置。第一次站到36个新生面前时，她紧张死了。两小时后，她们在讨论半夜去吃夜宵。"

10 个维度 = 找"过程"的 10 个角度。

### 实际效果
7 篇学生故事从每篇 3-5 行扩展到 800-1500 字叙事。输出可以直接做视频脚本素材。

## 模式五：agent 分工策略

### 什么时候用单个 agent
- 任务之间有关联（A 的输出影响 B 的输入）
- 内容量不大（<10 个文件要读）
- 需要统一的风格和判断标准

### 什么时候并行多个 agent
- 任务之间互不依赖
- 不同 agent 关注不同维度（如一个看数据、一个看故事、一个看引用）
- 内容量很大（几十到几百个文件）

### 并行的正确姿势

**先自己侦察再分配**：
```
1. 自己先快速浏览 5-10 个代表性文件 → 了解内容质量
2. 基于了解到的情况设计 agent 任务（知道该限定什么、该跳过什么）
3. 3 个 agent 并行工作
4. 自己汇总和连接结果
```

而不是："我不知道里面有什么，你们三个自己去看吧"← agent 会迷失。

### 📌 关键洞察

> **人做"判断"，AI 做"提取"。** 人在中间的角色是：看结果、判断好坏、决定下一步。不要让 AI 替你判断内容质量——它可以提醒你（"这里可能有问题"），但最终要用自己的业务直觉来定。

## 常见翻车和预防

| 翻车 | 原因 | 预防 |
|------|------|------|
| Agent 读了大量空文件浪费时间 | 没限定字数阈值 | 明确告诉 agent "只读 word_count > 200 的文件" |
| 提取的内容组织混乱 | 没给输出格式模板 | 在 prompt 里给出具体的输出结构 |
| 多个 agent 提取了重复内容 | 分工边界模糊 | 每个 agent 有明确的主题范围 + 关键词 |
| 文件之间信息矛盾 | 不同来源给了不同说法 | 在 prompt 里指定"遇到矛盾以[来源]为准" |
| 生成的内容太"AI 味" | 没给风格要求 | 明确要求"保留原话"、"不要改写直接引用" |
| 一次做太多返工 | 没先做小样确认 | 先做一个文件 → 确认方向对 → 再批量 |

---

## 总结：有效 AI 协作的核心公式

```
清晰的范围限定
+ 具体的输出格式
+ 一个先验证过的小样
+ 人做判断、AI 做提取
= 高质量的知识库内容
```

缺任何一环，要么返工，要么产出质量下降。
