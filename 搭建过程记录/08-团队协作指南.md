# 🤝 团队协作指南（搭建方法论）

> 本文档记录南桥 AI 自动化工作台**团队协作体系的设计过程**——为什么这样分角色、为什么用 Git 做强制隔离、onboarding 各阶段怎么设计、踩过什么坑。
>
> 📋 **日常操作手册**见根目录的 [团队协作指南.md](../团队协作指南.md)——那是给新成员看的简洁版说明书，这个文件是背后的完整方法论。

## 一、三种角色，泾渭分明

```
         ┌──────────────────────────────────────┐
         │         👤 Owner · Jess              │
         │   唯一掌握"事实"和"判断"的人          │
         │   修改知识库 · 审核产出 · 拍板决策     │
         └──────────────┬───────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼                               ▼
┌───────────────────┐          ┌───────────────────┐
│  🤖 Builder       │          │  📱 User           │
│  搭建者/协作者     │          │  使用者/员工       │
│                   │          │                   │
│  写 workflow      │          │  用 AI 查信息      │
│  调 prompt        │          │  用 AI 生成内容    │
│  搭自动化流程     │          │  用 AI 辅助客服    │
│                   │          │                   │
│  不能改知识库      │          │  完全不接触代码    │
│  产出进 drafts    │          │  不改任何文件      │
└───────────────────┘          └───────────────────┘
```

| 维度 | 👤 Owner | 🤖 Builder | 📱 User |
|------|----------|------------|---------|
| **典型身份** | Jess（老板） | AI 工程师协作者 | 客服、内容运营、实习生 |
| **核心任务** | 维护知识库、审核产出、做判断 | 搭 workflow/skill、调 prompt、优化产出质量 | 用 AI 辅助日常工作 |
| **需要的技能** | 行业知识 + AI 使用能力 | AI/工程能力（不需要行业知识） | 会用 Claude Code 或微信即可 |
| **Git 权限** | push 到 main、merge PR | 创建分支、提 PR | 不需要 Git |
| **修改知识库** | ✅ 唯一可以 | ❌ 禁止 | ❌ 禁止 |
| **修改 prompt** | ✅ | ✅（在自己的目录） | ❌ |
| **写 workflow** | ✅ | ✅（在自己的目录） | ❌ |
| **使用 AI 产出** | ✅ | ✅（测试用） | ✅（日常工作） |

## 二、目录权限表

### 核心原则：Builder 的产出不进知识库，先进"缓冲区"

```
自动化agent code/
│
├── 🏢 上纽昆杜升学辅导/          Owner: ✏️  Builder: 👁️  User: 👁️(通过AI)
├── 📚 昆杜官网资料/              Owner: ✏️  Builder: 👁️  User: 👁️(通过AI)
├── 📚 上纽官网资料/              Owner: ✏️  Builder: 👁️  User: 👁️(通过AI)
├── 📱 昆山杜克大学公众号资料/     Owner: ✏️  Builder: 👁️  User: 👁️(通过AI)
├── 📖 搭建过程记录/              Owner: ✏️  Builder: 👁️  User: —
│
├── 🤖 prompts/                  Owner: ✏️  Builder: ✏️  User: —
│   ├── jess/                    ← Owner 的 prompt 模板
│   └── collaborator/            ← Builder 的 prompt 模板（他们的工作区）
│
├── 🤖 workflows/                Owner: ✏️  Builder: ✏️  User: —
│   ├── jess/                    ← Owner 的 workflow
│   └── collaborator/            ← Builder 的 workflow
│
├── 📦 outputs/                  Owner: ✏️  Builder: ✏️  User: 👁️(使用)
│   ├── drafts/                  ← Builder 产出的初稿（待审核）
│   └── approved/                ← Owner 审核通过、可以发布的内容
│
└── 📦 原始数据（公众号txt/爬虫）  Owner: ✏️  Builder: —    User: —
```

**图例**：✏️ = 可以修改 · 👁️ = 只读（通过 AI 查询或直接看） · — = 不需要接触

### 关键规则

1. **Builder 的所有新 prompt 放 `prompts/collaborator/`**，不往现有的 prompt 模板库（`搭建过程记录/05-Prompt模板库.md`）里加。Owner 审核通过后，自己决定是否迁移。

2. **Builder 的所有 workflow/skill 放 `workflows/collaborator/`**，不和 Owner 的混放。

3. **AI 产出的脚本、图文全部进 `outputs/drafts/`**，不进知识库。Owner 审核通过后，手动把好的金句、角度提炼回知识库，再把可发布的移到 `outputs/approved/`。

4. **User 看不到 Builder 的工作区**，他们只通过 Claude Code 的 slash command 或微信调用已部署的 skill。

## 三、Git 分支策略（强制执行）

```
main ──────────────────────────────────────────────→
  │                                                   
  │  ← 只有 Owner 可以 push                          
  │  ← 知识库修改只发生在 main                        
  │                                                   
  ├── feature/xxx ────→ PR ──→ Owner review ──→ merge 
  │   (Builder A 的工作分支)                          
  │                                                   
  ├── feature/yyy ────→ PR ──→ Owner review ──→ merge 
  │   (Builder B 的工作分支)                          
```

### 规则

| 规则 | 说明 |
|------|------|
| **main 受保护** | 只有 Owner 可以直接 push |
| **Builder 在 feature 分支工作** | 分支名格式：`feature/功能描述` 或 `fix/修正内容` |
| **所有 Builder 的改动必须通过 PR** | 提交到 main 之前，Owner 在 PR 里 review |
| **知识库文件的 PR 必须 Owner approve** | 如果 Builder 的 PR 不小心改了知识库，直接拒绝 |
| **一个分支只做一件事** | 不要混着改 workflow 又改知识库 |

### GitHub / GitLab 设置（建议）

```
Branch protection for main:
  - Require a pull request before merging: ✅
  - Require approvals: 1
  - Dismiss stale reviews: ✅
  - Require review from Code Owners: ✅
  - Allow force pushes: ❌
  - Lock branch: ✅

CODEOWNERS 文件内容：
  上纽昆杜升学辅导/   @jess
  昆杜官网资料/       @jess
  上纽官网资料/       @jess
  昆山杜克大学公众号资料/ @jess
  搭建过程记录/       @jess
  项目总览.md         @jess
```

## 四、日常协作流程

### 4.1 Builder 的一天

```
1️⃣ 同步最新状态
   git checkout main
   git pull origin main
   git checkout -b feature/我的任务

2️⃣ 工作（只在自己的目录里）
   可以改：
   - prompts/collaborator/  ← 写 prompt 模板
   - workflows/collaborator/ ← 写 workflow 脚本
   - outputs/drafts/        ← AI 生成初稿放这里

   绝对不能改（改了 PR 也会被拒）：
   - 上纽昆杜升学辅导/
   - 昆杜官网资料/
   - 上纽官网资料/
   - 昆山杜克大学公众号资料/
   - 搭建过程记录/
   - 项目总览.md

3️⃣ 提交 & 提 PR
   git add prompts/collaborator/ workflows/collaborator/
   git commit -m "feat: 视频脚本三步生成 workflow"
   git push origin feature/我的任务
   → 在 GitHub 上创建 Pull Request

4️⃣ 等 Owner Review
   - Owner 会检查 prompt 逻辑和产出样本
   - 有问题的会 comment，需要修改
   - 审核通过后 Owner 合并到 main
```

### 4.2 Owner 的一天

```
1️⃣ 维护知识库（直接在 main 上）
   git checkout main
   # 修改知识库文件（新增学生故事、更新课表、调整品牌调性...）
   git add 上纽昆杜升学辅导/
   git commit -m "知识库更新：新增2026申请季的数据"
   git push origin main

2️⃣ Review Builder 的 PR
   - 打开 GitHub PR 页面
   - 检查：有没有改到不该改的目录?
   - 检查：prompt 逻辑对不对?
   - 检查：用几个选题跑一下，看 AI 产出质量
   - 通过 → Merge
   - 有问题 → Comment，Request Changes

3️⃣ 提炼好内容回知识库
   # 从 outputs/drafts/ 中审核通过的内容
   - 好的金句 → 加入金句库
   - 好的选题角度 → 加入选题库
   - 新的学生故事 → 加入过往案例
   
4️⃣ 更新计划看板
   # 编辑 搭建过程记录/07-日期追踪与计划看板.md
   - 标记完成的任务
   - 记录新发现的问题
```

### 4.3 User 的一天

```
User 不需要 Git，不需要知道项目结构。

方式一：在 Claude Code 里用 slash command
   /generate-video-script     ← 生成视频脚本
   /generate-xhs-post         ← 生成小红书图文
   /query-knowledge           ← 查知识库（如"昆杜的通识课程有哪些"）
   /customer-faq              ← 回答家长常见问题

方式二：通过微信远程调用
   发微信消息 → Claude Code 收到 → 执行任务 → 返回结果

User 只需要：
   1. 知道有哪些 slash command 可以用
   2. 会描述自己想要什么
   3. 把 AI 产出的内容复制到发布平台

User 不需要：
   - Git / GitHub 账号
   - 了解 prompt 怎么写
   - 了解知识库结构
   - 修改任何文件
```

## 五、Builder 上手路径（Onboarding）

一个新 Builder 加入时，按以下顺序引导。目标：**尽快能产出，但绝不污染知识库**。

### 阶段 1：理解项目（第 1 天，只读）

```
必读文件（按顺序）：
1. 项目总览.md                                  ← 知道整体是什么
2. 搭建过程记录/02-架构设计原则.md                ← 知道为什么这样设计
3. 搭建过程记录/05-Prompt模板库.md                ← 知道 AI 怎么被调用
4. 上纽昆杜升学辅导/06-品牌调性.md                 ← 知道我们对外怎么说话

选读（想做视频脚本再看）：
5. 昆山杜克大学公众号资料/01-内容全景总览.md
6. 上纽昆杜升学辅导/01-公司概览.md
```

> ⚠️ **不要**让 Builder 一开始就读全部 50+ 个文件。读得越多，他们越容易产生"我懂了"的错觉，然后写出错误的东西。只读 4 个核心文件就够了。

### 阶段 2：看示例（第 1-2 天，观察）

```
让 Builder 看一次完整流程：
1. Owner 在 Claude Code 里运行一次完整的脚本生成
2. Builder 观察整个 chain 的执行过程
3. Builder 看 outputs/approved/ 里审核通过的成品
4. Builder 对比 outputs/drafts/ 里被驳回的版本，理解差距
```

### 阶段 3：沙盒试手（第 2-3 天，受限写作）

```
给 Builder 第一个任务（安全等级 🟢）：
- "优化这个 prompt，让 hook 句更抓眼球"
- "写一个封面图生成的 HTML 模板"
- "把这三步串成一个 slash command"

Builder 只能在以下目录操作：
- prompts/collaborator/
- workflows/collaborator/
- outputs/drafts/

产出审核标准：
✅ prompt 结构合理
✅ 跑几次结果不跑偏
✅ 没有编造学校信息
❌ 只有 Owner 确认无误后，才允许进入下一阶段
```

### 阶段 4：独立产出（1 周后，持续 review）

```
Builder 可以独立接任务，但流程不变：
1. 在 feature 分支工作
2. 产出进 drafts
3. PR → Owner review → merge

高风险信号（Owner 需要注意）：
🚩 Builder 说"我觉得这个知识库文件应该改成..."
🚩 Builder 在 PR 里改了知识库目录的文件
🚩 Builder 的 prompt 里出现了他不可能知道的事实信息
🚩 生成的脚本里有"听起来对但实际不是"的内容
```

## 六、User 上手路径

User 不需要理解项目，只需要会用工具。

### 新手引导（30 分钟）

```
1. 打开 Claude Code（或发一条微信测试消息）
2. 学习 3 个最常用的命令：
   - /generate-video-script  ← 输入选题，生成脚本
   - /query-knowledge        ← 问任何学校/服务相关问题
   - /customer-faq           ← 回答家长问题
3. 尝试一次：说一个真实场景，让 AI 执行
4. 确认能独立操作
```

### User 权限边界

```
✅ User 可以做：
   - 调用已部署的 skill 和 workflow
   - 查询知识库内容
   - 生成内容并发布（审核流程另定）
   - 反馈"这个生成的不对"给 Owner

❌ User 不能做：
   - 修改任何文件
   - 创建新的 skill/workflow
   - 修改 prompt
   - 访问 Git
   - 直接编辑知识库
```

## 七、常见协作场景

### 场景 1：Builder 要测试 prompt，需要用到知识库

```
✅ 正确做法：
   Builder 把 prompt 写在 prompts/collaborator/ 里
   → 在自己的 Claude Code 里加载知识库测试
   → 知识库是只读引用，不会被他的 prompt 修改

❌ 错误做法：
   Builder 觉得"这段知识不够清楚"，直接改了知识库文件
   → PR 会被拒绝
```

### 场景 2：Builder 产出里有非常好的内容，Owner 想吸收

```
正确流程：
1. Owner 在 PR review 里标记好的部分
2. Merge PR（好的产出进入 outputs/drafts/ 或 workflows/）
3. Owner 手动提炼好的金句/角度 → 写回知识库
4. Owner 单独 commit："从协作产出提炼金句 ×3"

错误流程：
❌ 让 Builder 直接往知识库里加
❌ 不审核就直接发布
```

### 场景 3：User（客服）说 AI 回答得不对

```
1. User 反馈给 Owner（微信或口头）
2. Owner 检查：
   - 是知识库信息错了？ → Owner 修改知识库
   - 是 prompt 不够好？ → Owner 或 Builder 改 prompt
   - 是 User 问法不对？ → 教 User 换个问法
3. Owner 修改后通知 User："已修复，再试一次"
```

### 场景 4：多个 Builder 同时工作

```
Git 分支隔离：
   Builder A: feature/video-script-v2
   Builder B: feature/cover-image-gen

互不冲突（各自的目录不同）：
   A 改 prompts/collaborator/video-script.md
   B 改 workflows/collaborator/cover-gen.ts

如果改了同一个文件：
   Git 会标记冲突 → Owner 协调 → 手动 merge
```

## 八、故障处理

### Q: Builder 不小心改了知识库文件并提交了 PR，怎么办？

```
1. 在 PR review 里 Request Changes
2. 告诉 Builder revert 掉那个文件的修改：
   git checkout origin/main -- 上纽昆杜升学辅导/某文件.md
   git commit --amend
   git push --force
3. 如果已经 merge 了：Owner 在 main 上 revert 那个 commit
```

### Q: User 不知道怎么描述需求，AI 产出一团糟？

```
不是 User 的问题，是 prompt/skill 还不够好用。

Owner / Builder 需要：
1. 观察 User 的原始输入
2. 优化 prompt，让 AI 更"容错"
3. 给 User 一个简单的模板："你就这样问我..."
```

### Q: 新 Builder 一直产出错误内容，怎么办？

```
诊断：
- 是 prompt 技术不行？ → 给更具体的示例和模板
- 是对业务理解不够？ → 让他只做纯技术任务（封面图、格式转换等）
- 是态度问题？ → 换人

底线：
- 只要不改知识库，Builder 的"错误"只是浪费他自己的时间
- 知识库是防火墙，他的错误不会传播到整个项目
```

## 九、角色扩展（未来）

当团队扩大到更多角色时，按此框架扩展：

```
未来可能新增：

📊 数据分析角色
   - 权限：只读 outputs/approved/ + 知识库
   - 任务：分析哪种内容表现好，给 Owner 选题建议

🎨 设计角色
   - 权限：读写 workflows/collaborator/（封面图相关）
   - 任务：维护封面图模板和视觉规范

📝 内容审核角色
   - 权限：只读 outputs/drafts/ → 标记"可用/不可用"
   - 权限：不能直接改知识库，但可以给 Owner 提修改建议

核心逻辑不变：
知识库只有 Owner 能改 → Owner 是质量的最终守门人
```
