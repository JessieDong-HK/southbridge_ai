# 02 架构设计原则

> 记录知识库的架构决策——为什么这样设计，以及什么时候该用/不该用某种方案。

---

## 核心设计目标

这套知识库要同时服务三种使用者：

| 使用者 | 怎么用 | 需要什么 |
|------|------|------|
| **Jess（自己）** | 日常查阅、决策参考 | 快速定位某个信息 |
| **新员工/合作者** | 入职 onboarding | 可按顺序阅读、理解全貌 |
| **AI agents** | 被分配某个任务（如写一篇小红书、模拟面试） | 只加载跟任务相关的1-3个文件 |

三种使用者的需求有冲突——人需要连贯的阅读体验，AI 需要精准的按需加载。架构设计要同时满足两者。

---

## 决策一：Markdown 文件 vs 向量数据库 vs 一个大文档

### 最初的想法：向量数据库

最早的计划是搭 SQLite + Qdrant（向量数据库）+ SentenceTransformers 多语言 embedding 模型——一个完整的技术栈。想法是"把所有资料向量化 → 搜索 → AI 自动找最相关的段落"。

### 为什么放弃了

1. **规模不匹配**：业务知识库最终只有 ~24 个文件、几十 KB。向量数据库是为了百万级文档设计的。用火箭炮打蚊子。
2. **AI agent 本来就能读文件**：Claude Code 的 agent 可以直接 Read 指定文件。与其让 agent 搜到"最相关段落"，不如把文件拆到合适的粒度，让 agent 直接读整个文件。
3. **维护成本**：向量数据库需要维护 embedding 模型、处理更新同步。Markdown 文件直接用 Git 管理。
4. **可读性**：Markdown 文件人可以打开看。向量数据库里的 embedding 不行。

### 为什么不做一个大文档

一个独立的"全部知识.md"：
- AI agent 每次加载全部内容，浪费 context window
- 人很难找到特定信息
- 更新一个段落要改动整个文件

### 最终方案：Markdown 文件 + 目录结构 + cross-reference

- 每个文件一个独立主题
- 用目录树组织层级
- 文件之间用相对路径链接
- AI agent 按需加载 1-3 个文件

### 📌 决策原则

> **规模决定方案。** 几百 KB 的知识库 → Markdown 就够了。几十 MB → 考虑全文搜索。几 GB → 才需要向量数据库。
>
> **人也要能读。** 知识库不是只给机器用的——自己能打开看、能手动改，跟能用 Git 管理，这些都很重要。

## 决策二：编号体系（01-、02-、03-...）

### 为什么用数字编号而不是纯中文名

- **浏览器/编辑器按文件名排序**：01 → 02 → 03 天然呈现逻辑顺序
- **快速引用**："去看 03-服务项目/ai面试" 比 "去看服务项目里的AI面试那个文件" 快
- **扩展时不用重命名**：如果以后想加一个"03.5"，放在 03 和 04 之间就行

### 编号逻辑

| 编号 | 板块 | 原因 |
|------|------|------|
| 01 | 公司概览 | 全局了解 → 第一优先级 |
| 02 | 两校简介 | 业务基础知识 → 第二优先级 |
| 03 | 服务项目 | 核心产品 → 有子文件夹 |
| 04 | 课程大纲 | 服务的教案 → 跟 03 密切相关 |
| 05 | 过往案例 | 参考材料 |
| 06 | 品牌调性 | 创作规范 |
| 07 | 工作流程 | 操作手册 |
| 08 | 昆杜资源库 | 外部参考 → 关联到昆杜官网资料/ |
| 09 | 搭建方法论 | 元文档（就是这里） |

### 📌 决策原则

> **编号反映优先级和使用频率，不反映创建顺序。** 01-03 是日常被 AI agent 引用最多的。

## 决策三：什么时候拆文件、什么时候合并

### 拆成独立文件的条件

满足以下任一条件就拆：
- 不同的人/AI agent 在不同场景下需要（比如 AI 面试和真人面试是两个不同的 task）
- 内容超过 ~200 行（太长 agent 加载浪费 context）
- 逻辑上是一个独立主题，可以单独阅读

### 合并在一起的条件

满足以下条件就合：
- 内容强关联，分开反而不好理解（比如深度版vs标准版的对比如果不放在同一页就失去意义）
- 总内容量很少（<50行），不值得单独一个文件
- 多个小项属于同一类别（如多个学生案例放在 05-过往案例/README.md 一起）

### 实际例子

| 拆了 | 为什么 |
|------|------|
| ai面试.md / 真人面试.md / 文书辅导.md | 不同场景，不同 agent |
| 昆杜官网资料/ 的 8 个文件 | 不同面试问题需要不同板块 |

| 合了 | 为什么 |
|------|------|
| 05-过往案例/README.md（5个案例在一起） | 对比分析有价值 |
| 01-公司概览.md（公司介绍+团队+服务+价格） | 全局了解需要一起看 |

## 决策四：双文件夹架构

### 为什么建两个独立文件夹

```
上纽昆杜升学辅导/     ← 业务知识库（"我们做什么"）
昆杜官网资料/         ← 外部资源库（"DKU 是什么"）
```

而不是把昆杜资料放在上纽昆杜升学辅导的子文件夹里。

**理由**：
1. **来源不同**：业务知识是我们写的，官网资料是从外部提取的——更新频率和方式完全不同
2. **可替换性**：如果以后 DKU 官网更新了内容，只需要重新爬 → 替换昆杜官网资料/，不影响业务知识库
3. **可扩展性**：以后加上纽官网资料/，结构清晰，不会让主知识库越来越臃肿
4. **引用清晰**：从业务知识库链接到外部资源库时，`../昆杜官网资料/` 一眼就知道是外部来源

### 📌 决策原则

> **来源不同 → 物理隔离。** 自己写的和外部提取的分开放，这样任何一个的更新都不会干扰另一个。

---

## 决策五：cross-reference 体系

### 不是所有引用都做成链接

只在以下情况建链接：
- A 文件提到了 B 文件的核心内容（如"详见XX"）
- 两个文件经常被同一个 AI agent 连续加载
- 外部资源库 ↔ 主知识库之间的桥梁

### 实际链接网络

```
主知识库                        
01-公司概览 ←→ 03-服务项目/      
02-两校简介 ←→ 08-昆杜官方资源库  ←→  昆杜官网资料/
03-服务项目/ai面试 ←→ 08-昆杜官方资源库
04-课程大纲/ ←→ 03-服务项目/
```

### 📌 决策原则

> **AI agent 不会自己猜关联，需要显式指明。** "如果你需要XX信息，去看YY文件"这种显式指引比让 agent 自己搜索高效得多。

---

## 决策六：中文文件名

### 为什么用中文

- 使用者是中文母语
- 在文件系统中浏览时直观
- 编号前缀已经解决了排序问题

### 如果团队中有非中文母语者

可以改成中英双语文件名，如 `01-公司概览-Company-Overview.md`。但当前项目不需要。

---

## 总结：这套架构的核心思想

1. **小而专注的文件** — 每个文件只讲一件事
2. **用目录树代替搜索** — 好的命名和结构 > 搜索引擎
3. **显式连接** — 告诉 AI 去哪里找相关信息，不要让它猜
4. **物理隔离不同来源** — 自己的和外部来源的分开放
5. **匹配规模** — 几十个文件不需要向量数据库
