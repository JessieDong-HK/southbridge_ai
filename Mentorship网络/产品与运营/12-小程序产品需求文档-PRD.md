## 一、项目概述

**国际教育 Mentorship 网络**——连接有真实经历的名校学长学姐和迷茫的学生。

- 第一个切入点：**美本转学**（后续扩展更多赛道）
- 不是中介，是陪伴式 mentorship
- 半公益、community-driven

### 用户角色

| 角色 | 描述 |
|------|------|
| **学生（Mentee）** | 想走某条留学路但迷茫，需要过来人指导 |
| **Mentor** | 走过这条路、拿到真实结果的学长学姐 |
| **管理员（Jess）** | 策展 mentor、处理异常、看数据 |

---

## 二、平台形态

| 形态 | 优先级 | 原因 |
|------|--------|------|
| **微信小程序** | 🔴 主 | 所有 mentor 和学生都在微信上，零摩擦 |
| 网站（Web） | 🟡 辅 | 有比没有好，非第一版必需 |

### 小程序技术要点
- 微信原生框架或 Taro/uni-app 等跨端框架
- 微信支付集成
- 需要微信服务号或企业微信（用于通知推送）
- 腾讯会议 API（后期接入，MVP 手动建会）

**Mentor 卡片必须包含的字段**：

| 字段 | 说明 | 数据来源 |
|------|------|---------|
| 头像 | mentor 照片 | mentor 上传 |
| 姓名 | 中文名 + 英文名 | mentor 资料 |
| 路径摘要 | 一句话 from → to | profile 数据 |
| 亮点数据 | 1-2 个最有说服力的数字（如 3.98 GPA、5 段科研） | profile 数据 |
| 能力标签 | 该 mentor 勾选的能力标签（icon + 文字，只展示前 5-6 个，超出折叠） | 标签系统维度二 |
| 可约层级 | 💬 / 📋 / 🌱（展示 icon） | mentor 设置 |
| 活人标签 | 2-5 个小字标签，灰色/浅色，像便签纸贴在卡片底部 | mentor 自己填 |

**筛选器交互**：
- 能力标签以 **chips** 形式横排展示，点击选中/取消
- 选中后显示为高亮 pill，带 × 移除
- 支持多选（AND 逻辑）
- 显示每个 chip 对应的结果数量
- "清除"按钮一键重置
- 筛选后 mentor 列表实时更新

**排序**：默认按 session 完成数 / 评分降序（MVP 阶段可先按加入时间倒序）

**时间选择规则**：

- 时间段粒度：1 小时
- 展示当地时间 08:00-22:00
- 学生最少选 1 个，建议 ≥ 3 个（文案提示）
- 选中的时间段高亮

## 六、数据模型（核心字段）

### Mentor

```
{
  id: string
  name: string               // 中文名 + 英文名
  avatar: string             // 头像 URL
  track: string[]            // 赛道（维度一），如 ['美本转学', '美本申请']
  path_summary: string       // 路径摘要，如 "UC Davis → 宾大/康奈尔"
  highlight: string          // 亮点数据，如 "3.98 GPA · 5段科研"
  about: string              // 完整故事（800字以内，由Jess团队撰写）
  capabilities: string[]     // 能力标签（维度二）
  vibe_tags: string[]        // 活人标签（维度三），≤5个
  service_tiers: string[]    // 可约服务（维度四），如 ['💬', '📋', '🌱']
  is_active: boolean         // 是否展示（Jess可控制上下架）
  session_count: number      // 完成的 session 数
  rating: number             // 平均评分
  attendance_rate: number    // 出勤率
  created_at: datetime
}
```

### Student

```
{
  id: string
  wechat_openid: string      // 微信 openid
  name: string
  timezone: string           // 时区，如 'GMT+8'
  bookings: Booking[]
  created_at: datetime
}
```

### Booking

```
{
  id: string
  student_id: string
  mentor_id: string
  service_tier: string       // '💬' | '📋' | '🌱'
  amount: number             // 支付金额
  status: string             // 'pending_mentor_confirm' | 'time_confirmed' | 'completed' | 'cancelled' | 'escalated'
  student_time_slots: [      // 学生选的时间段（多个）
    { date: date, start: time, end: time }
  ]
  confirmed_time: datetime   // mentor 确认的时间（择一）
  meeting_link: string       // 腾讯会议链接（mentor确认后生成）
  student_note: string       // 学生填的简述
  mentor_no_response_since: datetime  // 用于升级判断
  created_at: datetime
}
```

## 八、开发建议

### 技术栈参考
- 小程序框架：微信原生 / Taro / uni-app
- 后端：Node.js / Python / Go（看开发者熟悉什么）
- 数据库：PostgreSQL 或 MySQL
- 文件存储：微信云开发 或 阿里云 OSS

### 开发顺序建议
1. **第一周**：Mentor 数据模型 + 赛道选择页 + Mentor 列表页 + 详情页（静态数据即可）
2. **第二周**：预约流程（选服务 → 选时间 → 简述 → 支付）+ 微信支付接入
3. **第三周**：学生端"我的"页面 + Mentor 确认（先做简单的，微信通知 + 手动确认）
4. **后续**：管理后台 + 通知系统 + 腾讯会议 API + 自动提醒

### 第一版可以用静态数据
前 3-5 个 mentor 的 profile 先 hardcode 或从 JSON 读取，不用搭后台。让展示和预约流程先跑通。
