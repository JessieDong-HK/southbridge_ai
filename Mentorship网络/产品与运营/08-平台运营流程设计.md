# 08 · 平台运营流程设计（交易→交付→复盘全链路）

> Jess 的核心判断（2026-06-18）：展示 mentor 信息不复杂。真正复杂的是**学生下单后怎么自动通知 mentor、怎么自动建会、怎么自动提醒**——这整条交付链路能不能做到不需要我插手。

---

## 一、平台形态选择

| 形态 | 优先级 | 原因 |
|------|--------|------|
| **微信小程序** | 🔴 主 | 所有 mentor 和客户都在微信上，零摩擦 |
| 网站 | 🟡 辅 | 有比没有好，但不是第一版必需 |

> 关键不是前端展示——展示老师信息、标签筛选、故事浏览，这些都不复杂。**复杂的是下面的自动化链路。**

---

## 二、完整用户流程

> ⚠️ 2026-06-18 Jess 修正：平台上的 mentor 都是愿意接单的——不是挑学生，是协调时间。对一小时短 session，大部分人改个时间就行。

### 2.1 短咨询 / 单次session 流程

```
【学生端】                          【系统】                         【Mentor端】
                                                                
在小程序浏览 mentor                                              
  → 看标签筛选                                                   
  → 读 profile 故事                                               
  → 选服务类型（¥99/599/2999…）                                   
  → 选时区                                                       
  → 勾选接下来一周内自己可以的                                      
    多个时间段（尽量多选）                                         
  → 下单 + 微信支付                                               
                     ───→  收到支付确认                           
                           把学生的多个时间选项                      ←  微信收到通知：
                           推送给对应 mentor                        "XX同学想约你，TA可以的
                            （专用微信号）                          时间有：A/B/C/D"
                                                                
                                                       ←  选一个具体时间
                                                          确认接单
                                                          或 → "这些时间都不行"
                     ───→  mentor 选了时间：                        
                           自动生成腾讯会议链接                      
                           通知双方：时间+会议号                    
                                                                
                     ───→  mentor 全拒：                           
                           通知学生："mentor 本周不可用，            
                           请重新选时间或换 mentor"                 
                                                                
                     ───→  mentor 超过 X 小时未回复：               
                           升级通知 → Jess                         
                           Jess 人工联系 mentor                    
                           （看是没看到通知/太忙/其他问题）           
                                                                
                     ───→  会议前 24h / 2h / 15min                
                           自动发送提醒 + 会议号                    
                           （双方都收到）                           
                                                                
学生+mentor 进入腾讯会议                                          
                     ───→  会议结束                                 
                           （Jess 可通过企业号后台                    
                            看到会议是否发生、时长）                  
                           自动推送课后反馈收集                      
                                                       ←  填写反馈
```

### 2.2 预约逻辑的关键设计

| 设计点 | 结论 | 原因 |
|--------|------|------|
| 谁先出时间 | **学生先多选**，mentor 再择一 | 降低 mentor 决策成本——选一个比想一个容易 |
| 选时间范围 | 接下来一周内 | 太远不确定，太近没空 |
| 最少选几个 | 建议 ≥ 3 个 | 提高 mentor 能匹配上的概率 |
| Mentor 拒单理由 | 只因为"这些时间都不行" | mentor 都愿意接，只是时间协调 |
| Mentor 不回复 | X 小时后升级到 Jess | Jess 人工判断：是没看到/太忙/出事了？ |

### 2.3 长线陪跑套餐（⚠️ 待解决）

> 产品 3/4（长期 mentorship + Elite 全包）涉及多次 session、可能多位 mentor 同时服务一个学生。这部分运营流程尚未设计。

**已知的复杂度**：
- 不是一次约一个时间，是长期协调 schedule
- 可能 Primary Mentor + Dream School Mentor + Writing Specialist 同时参与
- Mentor 之间需要信息同步（A mentor 和 B mentor 不能各说各的）
- 学生进度需要跨 mentor 可见

**当前状态**：先不做，单次 session 跑通后再设计。

## 三、核心模块拆解

### 模块 1｜通知系统（最关键）

**为什么不能用公众号模板消息**：
- 用户关掉公众号推送 = 消息石沉大海
- 模板消息有严格限制，触发条件受限

**方案：专用微信号**
- 一个独立的微信服务号/企业微信号，专门用来发通知
- 学生下单 → 这个号直接发微信消息给 mentor
- 提醒通知同理
- Mentor 不需要"打开 App 查看通知"——直接在微信聊天里看到，回复即确认

**通知节点**：

| 触发事件 | 通知对象 | 内容 |
|---------|---------|------|
| 学生下单并支付（含多个时间段） | Mentor | "XX 同学预约了你的 XX 服务。TA 可以的时间：A/B/C/D。请选一个。" |
| Mentor 选了时间 | 学生 | "XX mentor 已确认，时间：X月X日 XX:XX，会议链接：XXX" |
| Mentor 全部时间都不行 | 学生 + Jess | "XX mentor 本周这些时间都不行。请重新选择时间段或换一位 mentor" |
| Mentor 超过 X 小时未回复 | Jess | ⚠️ "XX mentor X 小时未回复 XX 同学的预约，请人工跟进" |
| 会议前 24h | 双方 | "明天 X 点有 mentorship session，会议号：XXX" |
| 会议前 2h | 双方 | "2 小时后有 mentorship session" |
| 会议前 15min | 双方 | "会议即将开始，点击加入：XXX" |
| 会议结束 | 学生 | "请对本次 session 进行反馈" |
| 会议结束 | Jess | "XX mentor + XX 学生 session 已完成（时长 XX min）" |

### 模块 2｜会议自动生成

**方案：腾讯会议企业版 API**
- 学生下单 + mentor 确认 → 自动调用腾讯会议 API 创建会议
- 自动设定会议时间、生成会议号和链接
- 自动发给双方

**为什么用企业号**：
- Jess 可以在企业号后台看到所有会议记录
- 知道每场会议是否发生、时长多少
- 不需要手动问"你们聊了吗"——系统有数据

### 模块 3｜时间协调机制（反向确认模式）

**核心前提**（Jess 的判断）：
- 在平台上的 mentor 都是愿意接单的，不会因为学生背景不好而拒单
- 唯一障碍是**时间**——但对 1 小时短 session，大部分人改个日期就行
- 所以不是"mentor 愿不愿意接"，而是"什么时间方便"

**流程设计**：

```
学生的责任：提供选项（多选）
  ↓
mentor 的责任：从中选一个（择一）
  ↓
如果都不行 → 学生重新选时间或换人
  ↓
如果不回复 → 升级到 Jess 人工处理
```

**具体步骤**：

1. **学生选时间**（支付前或支付后）
   - 系统根据学生时区显示接下来一周的空档
   - 学生勾选尽可能多的时间段（建议 ≥ 3 个）
   - "你选得越多，mentor 越容易匹配上"

2. **Mentor 收到选项**
   - 微信消息列出学生可以的 N 个时间段
   - Mentor 点一个确认 → 完成
   - 或点"这些时间都不行" → 触发重新选择

3. **Mentor 不回复怎么处理**

   | 时间线 | 动作 |
   |--------|------|
   | 0h | 学生下单，通知发出 |
   | X h 后（如 12h 或 24h） | 仍未回复 → 系统自动提醒 mentor 一次 |
   | X+? h 后 | 仍未回复 → ⚠️ 升级通知发给 Jess |
   | Jess 收到后 | 人工微信联系 mentor："看到系统给你推了 XX 学生的预约，是没有收到通知还是最近太忙？" |
   | 如果 mentor 确实太忙 | Jess 帮学生换人或换时间 |
   | 如果 mentor 多次不回复 | Jess 判断是否要从平台上暂时隐藏该 mentor |

   > 升级到 Jess 是必要的人工兜底——系统解决不了的，人来解决。但不回复应该是低频事件。

### 模块 3.1｜时区处理

- 学生在预约时设置自己的时区
- 系统展示的时间以学生时区为准
- 通知 mentor 时自动转换为 mentor 所在时区
- 会议链接（腾讯会议）自带时区显示

### 模块 4｜支付

**方案**：微信支付（小程序原生支持）
- 学生在小程序内完成支付
- 支付成功 = 订单确认
- 资金先到平台 → 再结算给 mentor

### 模块 5｜课后反馈收集

- 会议结束后自动推送简短反馈表
- 学生评分 + 一句话评价
- 反馈存入 mentor 档案 → 长期积累信任背书

## 四、技术实现难度评估

| 模块 | 复杂度 | 说明 |
|------|--------|------|
| 小程序前端（mentor 展示+筛选） | 🟢 低 | 标签筛选 + profile 页面，标准功能 |
| 微信支付 | 🟢 低 | 小程序原生，文档完善 |
| 学生多选时间 + 时区处理 | 🟡 中 | 交互逻辑稍复杂，但前端可实现 |
| 专用微信号通知 | 🟡 中 | 需要申请服务号或企业微信，有审核流程 |
| Mentor 择一确认 + 全拒 + 不回复升级 | 🟡 中 | 三种分支路径，需处理边界 |
| 腾讯会议 API 自动建会 | 🟡 中 | 需要企业版，有 API 文档 |
| 自动提醒（定时任务） | 🟢 低 | 标准的 cron job |
| 后台管理（Jess 看数据） | 🟢 低 | 简单的管理后台 |
| 长线陪跑套餐运营 | 🔴 高 | **待解决**——多次/多 mentor/长期协同 |

**MVP 建议**：先做小程序前端（浏览+下单+支付）+ 微信通知 + 手动建会，跑通 0→1 后再接腾讯会议 API 和全自动提醒。

## 五、MVP vs 完整版

| 功能 | MVP | 完整版 |
|------|-----|--------|
| Mentor 展示 + 标签筛选 | ✅ | ✅ |
| 学生多选时间段 + 时区 | ✅ | ✅ |
| 下单 + 微信支付 | ✅ | ✅ |
| 微信通知 mentor（时间选项） | ✅（半自动） | ✅（全自动） |
| Mentor 择一确认 | ✅（微信群/私聊回复） | ✅（系统内点选） |
| Mentor 不回复 → 升级 Jess | ❌（Jess 自己盯着） | ✅（自动升级） |
| 自动生成会议链接 | ❌（手动创建） | ✅（腾讯会议 API） |
| 自动提醒 | ❌（手动发微信） | ✅（定时推送 24h/2h/15min） |
| 课后反馈收集 | ❌（手动问） | ✅（自动推送） |
| Jess 后台看数据 | ✅（简单版） | ✅（完整仪表盘） |
| 长线陪跑套餐 | ❌ | ❌（待设计） |

> **MVP 核心**：先让 mentor 能被看到、学生能付钱、学生多选时间发给 mentor、mentor 能回复确认。其余的先手动，跑通以后再自动化。

## 八、待解决问题：长线陪跑套餐

> Jess 的判断（2026-06-18）：这部分现在完全不知道怎么搞。

### 问题描述

产品 3（长期 Fellowship Mentorship，¥9,900 / 6-18 月）和产品 4（Elite 全包，¥30,000）与短咨询有本质不同：

| 维度 | 短咨询（产品 1/2） | 长线陪跑（产品 3/4） |
|------|-------------------|---------------------|
| 次数 | 1 次 session | 多次，持续 6-18 个月 |
| Mentor 数 | 1 位 | 可能 2-4 位（Primary + Dream School + Writing + Research） |
| 排期 | 一次约一个时间 | 长期协调 schedule |
| 信息同步 | 不需要 | 多 mentor 之间需要知道彼此在做什么 |
| 进度追踪 | 不需要 | 学生进度需要跨 mentor 可见 |
| 交付管理 | 一次 session 完成即交付 | 长期推进，中间可能 timeline 崩塌、学生 burnout、换方向 |

### 已知的棘手问题

1. **多 mentor 怎么协同**：Primary Mentor 和 Dream School Mentor 和 Writing Specialist 各说各的怎么办？谁对学生的整体 narrative 负责？
2. **Schedule 怎么管**：不是"约一次下周的时间"，而是未来半年内持续发生的多次会议
3. **进度怎么追踪**：Jess 怎么远程知道一个长期学生现在走到哪了、有没有出问题
4. **如果 mentor 中途退出**：长期关系中 mentor 因个人原因不能继续了，怎么交接给新 mentor
5. **Mentor 之间信息壁垒**：A mentor 不知道 B mentor 跟学生聊了什么，可能导致建议矛盾

### 当前策略

**先不做。** 单次 session（产品 1/2）的预约-交付-反馈链路跑通后，再回过头来设计长线套餐的运营流程。
