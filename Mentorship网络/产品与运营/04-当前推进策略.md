# 04 · 当前推进策略（Jess · 2026-06-18）

## 整体逻辑

这个项目推进起来比较简单，核心就三件事：

1. **供给端**：不断扩充 mentor 数量 + 把每个人的 profile 和能提供的服务搞清楚
2. **平台端**：快速搭网站/小程序，让 mentor 可展示、学生可匹配
3. **流量端**：自媒体内容作为流量入口，与平台联动

**当前阶段重点：1 > 2 > 3。**

---

## 工作流 1｜Mentor 招募与 Onboarding 体系 🔴 最高优先

> 不管网站做成什么样，mentor 供给是前提。没有人，平台就是空壳。

### 1.1 项目介绍（PPT / PDF）

**目的**：发给潜在 mentor，让对方快速理解：
- 这是什么项目
- 为什么值得加入（四层价值：归属感、意义感、成长感、个人品牌）
- 我怎么参与（时间投入、服务边界、合作方式）
- 我能获得什么

**要求**：清晰、专业、有说服力——像一个正经的邀请，不是一个随意的微信消息。

### 1.2 深度采访提纲

**核心洞察**（Jess 实践经验）：

> 人物介绍如果交给 mentor 自己写，一定会非常敷衍，而且写不到点子上。

**正确做法**：
- 提前设计好一套问题框架——有点像记者采访前的准备稿
- 可以通过两种方式执行：
  - 方式 A：把问题发给 mentor，让他们自己填写
  - 方式 B：约一个线上 meeting，Jess 来问，实时深挖
- 收集到的回答由 Jess 团队润色成直达学生痛点的人物 profile

**问题设计的方向**（需要覆盖但不限于）：
- 成长路径：你从哪来、经历了什么、变成了什么样的人
- 关键转折：哪些 moment 真正塑造了你
- 痛点共鸣：你当年最迷茫/最崩溃的时候是什么样的
- 实战干货：你真正做对了什么？哪些是弯路？
- 我能帮你什么：你具体擅长辅导哪类学生、哪些问题

### 1.3 分级标签系统

**为什么需要标签系统**：

留学 mentorship 能做的事情太多太多了——选校、文书、科研、推荐信、面试、情绪支持、时间管理……不是每个人都愿意做所有事，也不是每个人都擅长所有事。

标签系统的目的：
1. **给 mentor 用**：快速勾选自己能做、愿意做的领域
2. **给学生用**：按需求筛选匹配的 mentor
3. **给平台用**：作为匹配算法的基础数据

**标签需要分级**：

```
一级标签（大类）         二级标签（细分能力）
─────────────────────  ─────────────────────
学业规划               GPA 策略、选课避坑、专业方向
科研背景提升           找 RA、Pioneer 申请、独立 research
推荐信布局              Office Hour 技巧、教授关系维护
文书辅导                Why Transfer、PS、supplemental essay
选校定专业              School list、fit 分析、专业匹配
面试辅导                Mock interview、自我介绍、学术对话
情绪与心态支持          焦虑疏导、burnout 应对、转学犹豫
校园融入               社交技巧、社团、文化适应
转学时间线管理          整体 timeline、阶段性 priority
……（持续补充）
```

**标签设计原则**：
- 一级标签覆盖大类，不超过 10-12 个
- 每个一级标签下 3-8 个二级标签
- Mentor 可以多选
- 区分"擅长"和"愿意做"（有些 mentor 擅长但不想做太多）

## 工作流 2｜小程序 / 网站搭建 + 运营自动化 🟡 中优先

> ⚠️ 2026-06-18 Jess 更新：展示层不复杂。真正复杂的是**交易后的交付链路**——怎么自动通知 mentor、自动建会、自动提醒、让我远程知道每次会议发生了什么。

### 平台形态：小程序优先

- **微信小程序** 🔴 主 —— 所有 mentor 和客户都在微信上，零摩擦
- **网站** 🟡 辅 —— 有比没有好，但不是第一版必需

### 2.1 展示层（简单）
- Mentor 列表 + 标签筛选 + 故事浏览 + 下单支付
- 标签系统已有 → [07-Mentor分级标签系统.md](07-Mentor分级标签系统.md)

### 2.2 运营自动化层（关键难点）

核心链路：**学生下单 → 微信通知 mentor → mentor 确认 → 自动建会 → 自动提醒 → 课后反馈**

关键设计决策：

| 决策 | 结论 | 原因 |
|------|------|------|
| 通知方式 | 专用微信号（非公众号模板消息） | 模板消息会被用户关掉 |
| 会议平台 | 腾讯会议企业版 API | 自动建会 + Jess 可远程看会议数据 |
| Mentor 确认 | 必须 mentor 主动确认才建会 | 不能假设 mentor 永远有空 |
| 提醒频率 | 会前 24h / 2h / 15min，共 3 次 | 包含会议号 |

> 完整设计见 [08-平台运营流程设计.md](08-平台运营流程设计.md)

### 2.3 MVP 分界线

**MVP 先做**：小程序前端（浏览+下单+支付）+ 微信通知（可半自动）+ 手动建会
**后续补齐**：腾讯会议 API 自动建会 + 全自动提醒 + 课后反馈自动收集

### 2.4 竞品对标扫描（依然需要）

快速搜索对标的 mentorship 小程序/网站，提取可复刻的设计元素。

> Echo（陈宇博）负责前端开发，6/19 完成 Horizon 交付后开工。

## 工作流 3｜自媒体与平台联动 🟢 低优先

**背景**：如果自媒体以多个切入点为内容策略（比如美本转学、海外高中、香港研究生…），那么在网站/小程序上是否应该有对应的快捷入口，让学生能快速找到相关 mentor 和内容。

**优先级低的原因**：
- 先把一个切入点（转学）跑通，再考虑多入口
- 网站第一版不需要完美，能用就行
- 后面完全可以加一个模块/页面
